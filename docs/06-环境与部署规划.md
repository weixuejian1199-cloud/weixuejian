## 一、环境分层策略

| 环境 | 用途 | 基础设施 | 数据 |
|------|------|---------|------|
| **local** | 本地开发调试 | Docker Compose 全部本地运行 | seed 数据 / mock 数据 |
| **staging** | 预发布验证 | 阿里云单节点，与生产同架构但资源缩小 | 脱敏的生产数据子集 |
| **production** | 正式生产 | 阿里云独立资源，数据库和缓存使用云托管服务 | 真实业务数据 |

### 环境隔离原则

1. **数据库完全隔离**：每个环境独立数据库实例，绝不共用
2. **配置独立**：每个环境独立 `.env` 文件，敏感信息生产环境使用 KMS
3. **网络隔离**：staging 和 production 在不同 VPC 或安全组中
4. **镜像统一**：所有环境使用同一份 Docker 镜像，仅通过环境变量区分行为

## 三、Docker Compose 配置规划

### 3.1 容器清单

| 容器 | 镜像 | 端口 | 说明 |
|------|------|------|------|
| `nginx` | nginx:alpine | 80, 443 | 反向代理 + HTTPS 终止 + 静态资源 |
| `backend` | 自建镜像 | 3000 | Node.js API 服务（Express） |
| `worker` | 自建镜像（同 backend） | - | BullMQ Worker，不暴露端口 |
| `postgres` | postgres:15-alpine | 5432 | 本地开发用，生产使用 RDS |
| `redis` | redis:7-alpine | 6379 | 本地开发用，生产使用云 Redis |
| `prometheus` | prom/prometheus | 9090 | 指标收集（Phase 1b） |
| `grafana` | grafana/grafana | 3001 | 监控面板（Phase 1b） |

### 3.2 docker-compose.yml（本地开发）

```yaml
version: "3.9"

services:
  # ─── 反向代理 ───
  nginx:
    image: nginx:alpine
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf:ro
      - ./nginx/ssl:/etc/nginx/ssl:ro
    depends_on:
      - backend
    restart: unless-stopped

  # ─── API 服务 ───
  backend:
    build:
      context: .
      dockerfile: Dockerfile
      target: development
    ports:
      - "3000:3000"
    env_file:
      - .env.local
    volumes:
      - ./src:/app/src
      - ./prisma:/app/prisma
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    command: pnpm dev
    restart: unless-stopped

  # ─── BullMQ Worker ───
  worker:
    build:
      context: .
      dockerfile: Dockerfile
      target: development
    env_file:
      - .env.local
    volumes:
      - ./src:/app/src
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    command: pnpm dev:worker
    restart: unless-stopped

  # ─── 数据库（仅本地） ───
  postgres:
    image: postgres:15-alpine
    ports:
      - "5432:5432"
    environment:
      POSTGRES_DB: enterprise_workstation
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: postgres_local_dev
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      timeout: 3s
      retries: 5
    restart: unless-stopped

  # ─── Redis（仅本地） ───
  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    command: redis-server --maxmemory 256mb --maxmemory-policy allkeys-lru
    volumes:
      - redisdata:/data
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 3s
      retries: 5
    restart: unless-stopped

volumes:
  pgdata:
  redisdata:
```

## 四、网络与安全

### 4.1 安全组规则

| 方向 | 端口 | 来源 | 用途 |
|------|------|------|------|
| 入站 | 80 | 0.0.0.0/0 | HTTP（重定向到 HTTPS） |
| 入站 | 443 | 0.0.0.0/0 | HTTPS |
| 入站 | 22 | 固定办公 IP | SSH 管理 |
| 入站 | 5432 | ECS 内网 IP | PostgreSQL（仅内网） |
| 入站 | 6379 | ECS 内网 IP | Redis（仅内网） |
| 入站 | 9090/3001 | ECS 内网 IP | Prometheus/Grafana（仅内网） |
| 出站 | 全部 | 0.0.0.0/0 | 允许所有出站 |

> **关键**：数据库和 Redis **严禁** 开放公网访问，只允许 ECS 所在安全组内网互通。

### 4.2 Nginx 完整配置方案

以下为生产环境 `nginx.prod.conf` 的完整配置方案（涵盖 SSL 终止、反向代理、安全头、日志格式等）：

```nginx
# /etc/nginx/nginx.prod.conf — 生产环境完整配置

user nginx;
worker_processes auto;
error_log /var/log/nginx/error.log warn;
pid /var/run/nginx.pid;

events {
    worker_connections 1024;
    multi_accept on;
}

http {
    include       /etc/nginx/mime.types;
    default_type  application/octet-stream;

    # ─── JSON 格式访问日志（方便日志服务采集） ───
    log_format json_combined escape=json
        '{'
            '"time":"$time_iso8601",'
            '"remote_addr":"$remote_addr",'
            '"request_method":"$request_method",'
            '"request_uri":"$request_uri",'
            '"status":$status,'
            '"body_bytes_sent":$body_bytes_sent,'
            '"request_time":$request_time,'
            '"upstream_response_time":"$upstream_response_time",'
            '"http_referer":"$http_referer",'
            '"http_user_agent":"$http_user_agent",'
            '"http_x_forwarded_for":"$http_x_forwarded_for",'
            '"request_id":"$request_id"'
        '}';

    access_log /var/log/nginx/access.log json_combined;

    # ─── 性能优化 ───
    sendfile on;
    tcp_nopush on;
    tcp_nodelay on;
    keepalive_timeout 65;
    types_hash_max_size 2048;

    # ─── Gzip 压缩 ───
    gzip on;
    gzip_vary on;
    gzip_proxied any;
    gzip_comp_level 6;
    gzip_min_length 1024;
    gzip_types
        text/plain
        text/css
        text/xml
        application/json
        application/javascript
        application/xml
        application/xml+rss
        image/svg+xml;

    # ─── 请求体大小限制 ───
    client_max_body_size 50m;

    # ─── SSL 全局配置 ───
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384;
    ssl_prefer_server_ciphers off;
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 1d;
    ssl_session_tickets off;

    # ─── 上游服务定义 ───
    upstream backend {
        server backend:3000;
        keepalive 32;
    }

    # ═══════════════════════════════════════════
    # HTTP → HTTPS 强制跳转（所有域名）
    # ═══════════════════════════════════════════
    server {
        listen 80;
        server_name api.cebrenis.com app.cebrenis.com cebrenis.com;
        return 301 https://$server_name$request_uri;
    }

    # ═══════════════════════════════════════════
    # api.cebrenis.com — API 服务
    # ═══════════════════════════════════════════
    server {
        listen 443 ssl http2;
        server_name api.cebrenis.com;

        ssl_certificate     /etc/nginx/ssl/fullchain.pem;
        ssl_certificate_key /etc/nginx/ssl/privkey.pem;

        # ─── 安全响应头 ───
        add_header X-Content-Type-Options nosniff always;
        add_header X-Frame-Options DENY always;
        add_header X-XSS-Protection "1; mode=block" always;
        add_header Referrer-Policy "strict-origin-when-cross-origin" always;
        add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;
        add_header Content-Security-Policy "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'" always;

        # ─── 隐藏 Nginx 版本号 ───
        server_tokens off;

        # ─── 请求 ID 透传 ───
        proxy_set_header X-Request-ID $request_id;

        # ─── API 反向代理 ───
        location /api/ {
            proxy_pass http://backend;
            proxy_http_version 1.1;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
            proxy_set_header Connection "";

            # SSE 流式响应支持（AI 对话 /api/v1/ai/chat）
            proxy_buffering off;
            proxy_cache off;
            proxy_read_timeout 300s;
        }

        # ─── WebSocket 升级（AI 对话实时推送） ───
        location /ws/ {
            proxy_pass http://backend;
            proxy_http_version 1.1;
            proxy_set_header Upgrade $http_upgrade;
            proxy_set_header Connection "upgrade";
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_read_timeout 86400s;
        }

        # ─── 健康检查端点（不记录日志） ───
        location = /api/v1/health {
            proxy_pass http://backend;
            access_log off;
        }

        # ─── 拒绝访问隐藏文件 ───
        location ~ /\. {
            deny all;
            access_log off;
            log_not_found off;
        }
    }

    # ═══════════════════════════════════════════
    # app.cebrenis.com — 小程序 H5 / 静态资源
    # ═══════════════════════════════════════════
    server {
        listen 443 ssl http2;
        server_name app.cebrenis.com;

        ssl_certificate     /etc/nginx/ssl/fullchain.pem;
        ssl_certificate_key /etc/nginx/ssl/privkey.pem;

        # 安全头（同上）
        add_header X-Content-Type-Options nosniff always;
        add_header X-Frame-Options SAMEORIGIN always;
        add_header Referrer-Policy "strict-origin-when-cross-origin" always;
        add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
        server_tokens off;

        root /usr/share/nginx/html;
        index index.html;

        # SPA 路由支持
        location / {
            try_files $uri $uri/ /index.html;
        }

        # 静态资源缓存
        location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff2?)$ {
            expires 30d;
            add_header Cache-Control "public, immutable";
        }
    }
}
```

**配置要点总结**：

| 特性 | 配置 | 说明 |
|------|------|------|
| SSL 终止 | TLS 1.2+ only | 禁用 TLS 1.0/1.1 |
| HTTPS 强制跳转 | 80 → 443 `return 301` | 所有 HTTP 请求重定向 |
| SSE 流式支持 | `proxy_buffering off` | AI 对话流式响应 |
| WebSocket | `Upgrade` + `Connection` 头 | 实时推送 |
| 请求大小 | `client_max_body_size 50m` | Excel/CSV 上传支持 |
| Gzip 压缩 | level 6，多种 MIME 类型 | 减少传输带宽 |
| 安全头 | X-Frame / X-Content-Type / HSTS / CSP / Referrer | 防 XSS/Clickjacking |
| 日志格式 | JSON 格式 | 方便 SLS/ELK 采集解析 |
| 请求 ID | `$request_id` 透传 | 全链路追踪 |

## 五、CI/CD 流水线

### 5.1 流水线总览

```
push to main ──→ lint + type-check + prisma validate ──→ vitest run ──→ build image ──→ push to ACR ──→ deploy to staging ──→ 健康检查
                                                                                                                                  │
                                                                                                                            ✅ 通过 → 完成
                                                                                                                            ❌ 失败 → 自动回滚

tag v*.*.* ────→ lint + type-check + prisma validate ──→ vitest run ──→ build image ──→ push to ACR ──→ deploy to production ──→ 健康检查
                                                                                                        (手动确认)                  │
                                                                                                                              ✅ 通过 → 完成
                                                                                                                              ❌ 失败 → 自动回滚
```

**触发条件**：

| 触发方式 | 条件 | 部署目标 |
|---------|------|---------|
| push | `main` 分支 | staging |
| tag | `v*.*.*` 格式 | production（需手动确认） |
| PR | 目标为 `main` | 仅运行检查和测试，不部署 |
| workflow_dispatch | 手动触发 | 可选 staging / production |

**流水线阶段**：

| 阶段 | 内容 | 失败处理 |
|------|------|---------|
| 1 - 检查 | `pnpm lint` + `pnpm tsc --noEmit` + `pnpm prisma validate` | 阻断流水线 |
| 2 - 测试 | `vitest run`（需 PostgreSQL + Redis service container） | 阻断流水线 |
| 3 - 构建 | `docker build --target production` + 推送到阿里云 ACR | 阻断流水线 |
| 4 - 部署 | SSH 到 ECS → `docker pull` → `docker compose up -d` | 触发回滚 |
| 5 - 验证 | 健康检查 `/api/v1/health`，30s 内通过 | 触发回滚 |

### 5.3 GitHub Actions Workflow

```yaml
# .github/workflows/deploy.yml

name: Build & Deploy

on:
  push:
    branches: [main]
    tags: ["v*.*.*"]
  pull_request:
    branches: [main]
  workflow_dispatch:
    inputs:
      target:
        description: "部署目标"
        required: true
        type: choice
        options:
          - staging
          - production

env:
  REGISTRY: registry.cn-hangzhou.aliyuncs.com
  IMAGE_NAME: enterprise-workstation/backend

jobs:
  # ─── 阶段 1：代码质量检查 ───
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v2
        with:
          version: 10
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: pnpm
      - run: pnpm install --frozen-lockfile
      - run: pnpm lint
      - run: pnpm tsc --noEmit           # TypeScript 类型检查
      - run: pnpm prisma validate         # Prisma Schema 校验
      - run: pnpm prisma generate         # 生成 Prisma Client（测试需要）

  # ─── 阶段 2：测试（需要 PostgreSQL + Redis） ───
  test:
    needs: check
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:15-alpine
        env:
          POSTGRES_DB: test_db
          POSTGRES_USER: postgres
          POSTGRES_PASSWORD: postgres
        ports: ["5432:5432"]
        options: --health-cmd pg_isready --health-interval 10s --health-timeout 5s --health-retries 5
      redis:
        image: redis:7-alpine
        ports: ["6379:6379"]
        options: --health-cmd "redis-cli ping" --health-interval 10s --health-timeout 5s --health-retries 5
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v2
        with:
          version: 10
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: pnpm
      - run: pnpm install --frozen-lockfile
      - run: pnpm prisma generate
      - run: pnpm prisma migrate deploy   # 应用迁移到测试数据库
        env:
          DATABASE_URL: postgresql://postgres:postgres@localhost:5432/test_db
      - run: pnpm vitest run --coverage
        env:
          DATABASE_URL: postgresql://postgres:postgres@localhost:5432/test_db
          REDIS_URL: redis://localhost:6379/0
          NODE_ENV: test

  # ─── 阶段 3：构建并推送镜像（PR 不执行） ───
  build-and-push:
    needs: test
    if: github.event_name != 'pull_request'
    runs-on: ubuntu-latest
    outputs:
      image_tag: ${{ steps.meta.outputs.tag }}
      prev_tag: ${{ steps.prev.outputs.tag }}
    steps:
      - uses: actions/checkout@v4

      - name: Login to ACR
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ secrets.ACR_USERNAME }}
          password: ${{ secrets.ACR_PASSWORD }}

      - name: Get previous image tag (for rollback)
        id: prev
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.STAGING_HOST }}
          username: ${{ secrets.STAGING_USER }}
          key: ${{ secrets.STAGING_SSH_KEY }}
          script: |
            cd /opt/enterprise-workstation
            cat .current-image-tag 2>/dev/null || echo "none"

      - name: Extract metadata
        id: meta
        run: |
          SHA=$(git rev-parse --short HEAD)
          if [[ "$GITHUB_REF" == refs/tags/v* ]]; then
            VERSION=${GITHUB_REF#refs/tags/}
            TAG="${VERSION}-${SHA}"
          else
            TAG="staging-${SHA}"
          fi
          echo "tag=${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${TAG}" >> $GITHUB_OUTPUT

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: |
            ${{ steps.meta.outputs.tag }}
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:latest
          target: production
          cache-from: type=registry,ref=${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:latest
          cache-to: type=inline

  # ─── 阶段 4a：部署到 staging ───
  deploy-staging:
    needs: build-and-push
    if: github.ref == 'refs/heads/main' || (github.event_name == 'workflow_dispatch' && github.event.inputs.target == 'staging')
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to staging
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.STAGING_HOST }}
          username: ${{ secrets.STAGING_USER }}
          key: ${{ secrets.STAGING_SSH_KEY }}
          script: |
            set -euo pipefail
            cd /opt/enterprise-workstation

            # 记录当前镜像用于回滚
            cat .current-image-tag 2>/dev/null > .prev-image-tag || true
            echo "${{ needs.build-and-push.outputs.image_tag }}" > .current-image-tag

            # 拉取新镜像并启动
            export IMAGE_TAG="${{ needs.build-and-push.outputs.image_tag }}"
            docker compose -f docker-compose.yml -f docker-compose.prod.yml pull
            docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d --remove-orphans

            # 健康检查：30s 内必须通过，每 5s 检查一次
            for i in $(seq 1 6); do
              if curl -sf http://localhost:3000/api/v1/health > /dev/null 2>&1; then
                echo "Health check passed on attempt $i"
                docker image prune -f
                exit 0
              fi
              echo "Health check attempt $i failed, retrying in 5s..."
              sleep 5
            done

            # 健康检查失败 → 自动回滚
            echo "Health check failed! Rolling back..."
            PREV_TAG=$(cat .prev-image-tag 2>/dev/null || echo "")
            if [ -n "$PREV_TAG" ]; then
              export IMAGE_TAG="$PREV_TAG"
              docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d --remove-orphans
              cp .prev-image-tag .current-image-tag
              echo "Rolled back to $PREV_TAG"
            fi
            exit 1

  # ─── 阶段 4b：部署到 production（仅 tag 触发或手动，需确认） ───
  deploy-production:
    needs: build-and-push
    if: startsWith(github.ref, 'refs/tags/v') || (github.event_name == 'workflow_dispatch' && github.event.inputs.target == 'production')
    runs-on: ubuntu-latest
    environment: production  # GitHub Environment Protection Rules：需要审批人确认
    steps:
      - name: Deploy to production
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.PRODUCTION_HOST }}
          username: ${{ secrets.PRODUCTION_USER }}
          key: ${{ secrets.PRODUCTION_SSH_KEY }}
          script: |
            set -euo pipefail
            cd /opt/enterprise-workstation

            # 记录当前镜像用于回滚
            cat .current-image-tag 2>/dev/null > .prev-image-tag || true
            echo "${{ needs.build-and-push.outputs.image_tag }}" > .current-image-tag

            # 数据库迁移（production 使用 migrate deploy，不会自动创建新迁移）
            docker compose -f docker-compose.yml -f docker-compose.prod.yml run --rm backend \
              npx prisma migrate deploy

            # 拉取新镜像并启动
            export IMAGE_TAG="${{ needs.build-and-push.outputs.image_tag }}"
            docker compose -f docker-compose.yml -f docker-compose.prod.yml pull
            docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d --remove-orphans

            # 健康检查：60s 内必须通过，每 5s 检查一次
            for i in $(seq 1 12); do
              if curl -sf http://localhost:3000/api/v1/health > /dev/null 2>&1; then
                echo "Health check passed on attempt $i"
                docker image prune -f
                exit 0
              fi
              echo "Health check attempt $i failed, retrying in 5s..."
              sleep 5
            done

            # 健康检查失败 → 自动回滚
            echo "Health check failed! Rolling back..."
            PREV_TAG=$(cat .prev-image-tag 2>/dev/null || echo "")
            if [ -n "$PREV_TAG" ]; then
              export IMAGE_TAG="$PREV_TAG"
              docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d --remove-orphans
              cp .prev-image-tag .current-image-tag
              echo "Rolled back to $PREV_TAG"
            fi
            exit 1
```

### 5.4 部署回滚机制

| 场景 | 回滚方式 | 说明 |
|------|---------|------|
| 健康检查失败 | **自动回滚**：流水线自动切回 `.prev-image-tag` 记录的上一版本镜像 | 无需人工干预 |
| 业务逻辑错误（上线后发现） | **手动回滚**：`cd /opt/enterprise-workstation && export IMAGE_TAG=$(cat .prev-image-tag) && docker compose up -d` | SSH 到服务器执行 |
| 数据库迁移问题 | **手动处理**：Prisma 不支持自动回退迁移，需编写反向 SQL 并手动执行 | 生产迁移前必须在 staging 验证 |

> **关键**：每次部署前将当前镜像 tag 写入 `.prev-image-tag`，确保始终有回滚目标。保留最近 5 个镜像版本在 ACR，不立即删除。

### 5.5 部署前检查清单

| 检查项 | staging | production |
|--------|---------|-----------|
| 所有测试通过 | 自动 | 自动 |
| Prisma migration 无冲突 | 自动 | 自动 |
| 镜像构建成功 | 自动 | 自动 |
| 环境变量已更新 | 手动确认 | 手动确认 |
| 数据库迁移已执行 | 自动（部署脚本） | 手动确认后执行 |
| 回滚方案就绪 | - | 手动确认 |

```yaml
global:
```

```yaml
  group_interval: 5m
```

## 九、日志聚合方案

### 9.1 日志分层演进

| 阶段 | 方案 | 说明 |
|------|------|------|
| **Phase 1a（当前）** | pino → JSON 文件 → `docker logs` 查看 | 最简方案，够用 |
| **Phase 1b** | pino → JSON → 阿里云日志服务 SLS（Logtail 采集） | 支持搜索、告警、可视化 |
| **Phase 2+** | 考虑 ELK 或继续 SLS | 取决于日志量和成本 |

### 9.2 日志分类

| 日志类型 | 文件/路径 | 内容 | 保留周期 |
|---------|----------|------|---------|
| **Access Log** | Nginx `access.log`（JSON 格式） | 请求方法/URI/状态码/延迟/UA | 30 天 |
| **Error Log** | Nginx `error.log` + 应用 `stderr` | 异常堆栈、5xx 错误 | 90 天 |
| **Audit Log** | 应用内 `audit.log` | 登录/权限变更/敏感操作/数据导出 | 180 天 |
| **AI Conversation Log** | 应用内 `ai.log` | AI 对话上下文、Token 消耗、模型选择 | 90 天 |

### 9.3 敏感数据脱敏

日志输出前自动脱敏以下字段，防止敏感信息泄露：

| 数据类型 | 原始值示例 | 脱敏后 | 实现方式 |
|---------|----------|--------|---------|
| 手机号 | `13812345678` | `138****5678` | pino redact + 正则替换 |
| 身份证号 | `310101199001011234` | `3101**********1234` | pino redact + 正则替换 |
| 银行卡号 | `6222021234567890` | `6222****7890` | pino redact + 正则替换 |
| API Key | `sk-ant-xxxx` | `sk-ant-****` | pino redact 配置 |
| 密码字段 | `password123` | `[REDACTED]` | pino redact paths |

**pino 脱敏配置示例**：

```typescript

  // 自定义序列化器处理手机号等

    req: (req) => ({
      method: req.method,

```

## 十、高可用演进路径

### 10.1 演进阶段

| 阶段 | 架构 | 单点风险 | 预估 RTO | 适用场景 |
|------|------|---------|---------|---------|
| **Phase 1a（当前）** | 单 ECS + Docker Compose | 全部单点 | 1-4 小时（手动恢复） | 内部用户 <100，接受短暂不可用 |
| **Phase 1b** | 单 ECS + RDS 高可用版 + Redis Sentinel | 数据库/Redis 高可用 | 15-30 分钟 | 买家用户增长，数据不能丢 |
| **Phase 2** | 2 ECS + Nginx 负载均衡 + RDS 高可用 | 后端双实例 | <5 分钟 | DAU >500，需要零停机部署 |
| **Phase 3** | 阿里云 ACK（K8s）或 Docker Swarm | 全面高可用 | <1 分钟 | DAU >5000，需要弹性伸缩 |

### 10.2 Phase 1b 高可用升级方案

```

                    │   Nginx      │
                    │  (单实例)     │

                           │
```

**关键升级动作**：

| 组件 | 当前 | 升级为 | 费用变化 | 说明 |
|------|------|--------|---------|------|
| PostgreSQL | RDS 基础版 1核2G | RDS **高可用版** 2核4G | +¥300/月 | 主备自动切换，RPO=0 |
| Redis | 云 Redis 标准版 1GB | 云 Redis **哨兵版** 1GB | +¥60/月 | 自动故障转移 |
| 备份 | 每日全量 | 每日全量 + **WAL 连续归档** | +¥10/月（OSS） | RPO 从 24h 降到 <1h |

### 10.3 Phase 2 零停机部署方案

```
                    ┌──────────────┐
                    │   Nginx      │
                    │  负载均衡     │
                    └──────┬───────┘
                           │
                ┌──────────┼──────────┐
                │                     │
         ┌──────┴───────┐     ┌──────┴───────┐
         │  Backend #1   │     │  Backend #2   │

         └───────────────┘     └───────────────┘
```

**Rolling Update 流程**：
1. 拉取新镜像到 ECS-2
2. ECS-2 启动新版本容器，通过健康检查
3. Nginx 将流量切到 ECS-2
4. ECS-1 停止旧容器，启动新版本
5. Nginx 恢复双节点负载均衡
6. 全程零停机

## 十一、容量规划

### 11.1 Phase 1 预估

| 维度 | 预估值 | 依据 |
|------|--------|------|
| 内部用户（员工/管理员） | <100 | 时皙团队规模 |
| 买家用户 | <1,000 | 初期运营规模 |
| API QPS（峰值） | <50 | 内部管理操作为主 |
| 数据库大小 | <10GB | 146 万用户 + 95 万订单数据在 ztdy-open，本地仅存元数据 |
| Redis 内存 | <1GB | Token 缓存 + BullMQ 队列 + 限流计数器 |
| AI 对话量 | <200 次/天 | 管理操作 + ACI 客服审核 |
| 文件存储（OSS） | <5GB/月 | 报表导出 + 备份 |

### 11.2 推荐资源规格

| 资源 | 规格 | 说明 |
|------|------|------|
| ECS | **ecs.c6.large**（2核4G） | CPU 密集型实例，足够 Phase 1 全部容器 |
| RDS PostgreSQL | 基础版 **2核4G 100GB SSD** | 比初始 1核2G 稍大，避免频繁升级 |
| 云 Redis | 标准版 **1GB** | 足够 Phase 1，80% 时再升级 |
| OSS | 按量付费 | 备份 + 文件存储 |
| 带宽 | 5Mbps 按流量计费 | API 为主，流量不大 |

### 11.3 扩容触发条件

| 监控指标 | 触发阈值 | 持续时间 | 扩容动作 |
|---------|---------|---------|---------|
| CPU 使用率 | >70% | 持续 3 天 | ECS 升级到 4核8G |
| 内存使用率 | >80% | 持续 1 天 | ECS 升级到 4核8G |
| 磁盘使用率 | >75% | 即时 | 扩容磁盘或清理日志 |
| 数据库 CPU | >70% | 持续 1 周 | RDS 升级到高可用版 2核4G |
| 数据库连接数 | >80% max | 持续 1 天 | 检查连接池配置，必要时升级 |
| Redis 内存 | >80% | 持续 1 天 | 升级到 2GB |
| API P95 延迟 | >500ms | 持续 3 天 | 分析瓶颈，考虑拆分服务 |

> **原则**：不预估不提前买。监控告警触发 → 分析瓶颈 → 精准扩容。避免过度配置浪费预算。

### 12.6 备份脚本完整逻辑

```bash
#!/bin/bash
# scripts/backup.sh — 每日凌晨 2 点全量备份 + Redis 快照 + 结果通知
# crontab: 0 2 * * * /opt/enterprise-workstation/scripts/backup.sh

set -euo pipefail

# ─── 环境变量 ───
DATE=$(date +%Y%m%d)
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
DUMP_FILE="/tmp/backup/${DATE}.dump"
DUMP_GZ="/tmp/backup/${DATE}.dump.gz"
LOG_FILE="/var/log/backup/${DATE}.log"
RETENTION_DAYS=30

mkdir -p /tmp/backup /var/log/backup

# ─── 通知函数 ───
notify_feishu() {
  local status=$1
  local message=$2
  curl -s -X POST "$FEISHU_ALERT_WEBHOOK_URL" \
    -H "Content-Type: application/json" \
    -d "{\"msg_type\":\"text\",\"content\":{\"text\":\"[备份${status}] ${message}\"}}" \
    > /dev/null 2>&1 || true
}

echo "[${TIMESTAMP}] Starting backup..." | tee -a "$LOG_FILE"

# ─── 1. PostgreSQL 全量备份 ───
echo "Step 1: pg_dump..." | tee -a "$LOG_FILE"
pg_dump -Fc -h "$DB_HOST" -U "$DB_USER" -d enterprise_workstation > "$DUMP_FILE" 2>> "$LOG_FILE"
gzip "$DUMP_FILE"
DUMP_SIZE=$(du -h "$DUMP_GZ" | cut -f1)
echo "  Dump size: ${DUMP_SIZE}" | tee -a "$LOG_FILE"

# ─── 2. 上传 PostgreSQL 备份到 OSS ───
echo "Step 2: Upload to OSS..." | tee -a "$LOG_FILE"
ossutil cp "$DUMP_GZ" "oss://ew-backup-prod/daily/${DATE}.dump.gz" \
  --access-key-id "$OSS_ACCESS_KEY_ID" \
  --access-key-secret "$OSS_ACCESS_KEY_SECRET" \
  --endpoint "$OSS_ENDPOINT" 2>> "$LOG_FILE"

# ─── 3. Redis RDB 快照上传 ───
echo "Step 3: Redis snapshot..." | tee -a "$LOG_FILE"
redis-cli -h "$REDIS_HOST" -a "$REDIS_PASSWORD" BGSAVE 2>> "$LOG_FILE"
sleep 10  # 等待 BGSAVE 完成
ossutil cp /data/dump.rdb "oss://ew-backup-prod/redis/${DATE}-dump.rdb" \
  --access-key-id "$OSS_ACCESS_KEY_ID" \
  --access-key-secret "$OSS_ACCESS_KEY_SECRET" \
  --endpoint "$OSS_ENDPOINT" 2>> "$LOG_FILE"

# ─── 4. 清理过期本地文件 ───
echo "Step 4: Cleanup..." | tee -a "$LOG_FILE"
find /tmp/backup -name "*.dump.gz" -mtime +${RETENTION_DAYS} -delete 2>> "$LOG_FILE"

# ─── 5. 验证备份完整性 ───
echo "Step 5: Verify..." | tee -a "$LOG_FILE"
ossutil stat "oss://ew-backup-prod/daily/${DATE}.dump.gz" \
  --access-key-id "$OSS_ACCESS_KEY_ID" \
  --access-key-secret "$OSS_ACCESS_KEY_SECRET" \
  --endpoint "$OSS_ENDPOINT" > /dev/null 2>> "$LOG_FILE"

if [ $? -eq 0 ]; then
  echo "[$(date)] Backup completed successfully: ${DUMP_SIZE}" | tee -a "$LOG_FILE"
  notify_feishu "成功" "${DATE} 备份完成，大小 ${DUMP_SIZE}"

  # 更新备份成功时间戳（供 Prometheus 告警规则使用）
  date +%s > /tmp/backup/last_success_timestamp
else
  echo "[$(date)] Backup verification FAILED!" | tee -a "$LOG_FILE"
  notify_feishu "失败" "${DATE} 备份验证失败，请立即检查！"
  exit 1
fi
```

### 12.7 恢复流程

```bash
# ─── 从每日全量备份恢复 ───

# 1. 从 OSS 下载备份
ossutil cp oss://ew-backup-prod/daily/20260329.dump.gz /tmp/restore.dump.gz
gunzip /tmp/restore.dump.gz

# 2. 恢复到目标数据库
pg_restore -h $DB_HOST -U $DB_USER -d enterprise_workstation -c /tmp/restore.dump

# 3. 验证数据完整性
psql -h $DB_HOST -U $DB_USER -d enterprise_workstation -c "
  SELECT 'tenants' as table_name, count(*) FROM tenants
  UNION ALL
  SELECT 'users', count(*) FROM users
  UNION ALL
  SELECT 'ai_conversations', count(*) FROM ai_conversations;
"

# ─── Redis 恢复 ───
# 1. 停止 Redis 服务
# 2. 从 OSS 下载 RDB 文件
ossutil cp oss://ew-backup-prod/redis/20260329-dump.rdb /data/dump.rdb
# 3. 启动 Redis 服务（自动加载 RDB）
```

### 12.8 恢复演练制度

| 项目 | 要求 |
