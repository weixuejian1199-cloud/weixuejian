## 一、环境分层策略

| 环境 | 用途 | 基础设施 | 数据 |
|------|------|---------|------|
| **local** | 本地开发调试 | Docker Compose 全部本地运行 | seed 数据 / mock 数据 |
| **staging** | 预发布验证 | 阿里云单节点，与生产同架构但资源缩小 | 脱敏的生产数据子集 |
| **production** | 正式生产 | 阿里云独立资源，数据库和缓存使用云托管服务 | 真实业务数据 |

### 环境隔离原则

1. **数据库完全隔离**：每个环境独立数据库实例，绝不共用
2. **配置独立**：每个环境独立 `.env` 文件，敏感信息生产环境使用 KMS
3. **网络隔离**：staging 和 production 在不同 VPC 或安全组中
4. **镜像统一**：所有环境使用同一份 Docker 镜像，仅通过环境变量区分行为

## 三、Docker Compose 配置规划

### 3.1 容器清单

| 容器 | 镜像 | 端口 | 说明 |
|------|------|------|------|
| `nginx` | nginx:alpine | 80, 443 | 反向代理 + HTTPS 终止 + 静态资源 |
| `backend` | 自建镜像 | 3000 | Node.js API 服务（Express） |
| `worker` | 自建镜像（同 backend） | - | BullMQ Worker，不暴露端口 |
| `postgres` | postgres:15-alpine | 5432 | 本地开发用，生产使用 RDS |
| `redis` | redis:7-alpine | 6379 | 本地开发用，生产使用云 Redis |
| `prometheus` | prom/prometheus | 9090 | 指标收集（Phase 1b） |
| `grafana` | grafana/grafana | 3001 | 监控面板（Phase 1b） |

### 3.2 docker-compose.yml（本地开发）

```yaml
version: "3.9"

services:
  # ─── 反向代理 ───
  nginx:
    image: nginx:alpine
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf:ro
      - ./nginx/ssl:/etc/nginx/ssl:ro
    depends_on:
      - backend
    restart: unless-stopped

  # ─── API 服务 ───
  backend:
    build:
      context: .
      dockerfile: Dockerfile
      target: development
    ports:
      - "3000:3000"
    env_file:
      - .env.local
    volumes:
      - ./src:/app/src
      - ./prisma:/app/prisma
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    command: pnpm dev
    restart: unless-stopped

  # ─── BullMQ Worker ───
  worker:
    build:
      context: .
      dockerfile: Dockerfile
      target: development
    env_file:
      - .env.local
    volumes:
      - ./src:/app/src
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    command: pnpm dev:worker
    restart: unless-stopped

  # ─── 数据库（仅本地） ───
  postgres:
    image: postgres:15-alpine
    ports:
      - "5432:5432"
    environment:
      POSTGRES_DB: enterprise_workstation
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: postgres_local_dev
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      timeout: 3s
      retries: 5
    restart: unless-stopped

  # ─── Redis（仅本地） ───
  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    command: redis-server --maxmemory 256mb --maxmemory-policy allkeys-lru
    volumes:
      - redisdata:/data
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 3s
      retries: 5
    restart: unless-stopped

volumes:
  pgdata:
  redisdata:
```

## 四、网络与安全

### 4.1 安全组规则

| 方向 | 端口 | 来源 | 用途 |
|------|------|------|------|
| 入站 | 80 | 0.0.0.0/0 | HTTP（重定向到 HTTPS） |
| 入站 | 443 | 0.0.0.0/0 | HTTPS |
| 入站 | 22 | 固定办公 IP | SSH 管理 |
| 入站 | 5432 | ECS 内网 IP | PostgreSQL（仅内网） |
| 入站 | 6379 | ECS 内网 IP | Redis（仅内网） |
| 入站 | 9090/3001 | ECS 内网 IP | Prometheus/Grafana（仅内网） |
| 出站 | 全部 | 0.0.0.0/0 | 允许所有出站 |

> **关键**：数据库和 Redis **严禁** 开放公网访问，只允许 ECS 所在安全组内网互通。

### 4.2 Nginx 配置要点

```nginx
# /etc/nginx/nginx.conf 核心片段

# gzip 压缩
gzip on;
gzip_types text/plain application/json application/javascript text/css;
gzip_min_length 1024;

# HTTPS 重定向
server {
    listen 80;
    server_name api.cebrenis.com;
    return 301 https://$server_name$request_uri;
}

# API 反向代理
server {
    listen 443 ssl http2;
    server_name api.cebrenis.com;

    ssl_certificate     /etc/nginx/ssl/fullchain.pem;
    ssl_certificate_key /etc/nginx/ssl/privkey.pem;
    ssl_protocols       TLSv1.2 TLSv1.3;

    # 安全头
    add_header X-Content-Type-Options nosniff;
    add_header X-Frame-Options DENY;
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

    # API 代理
    location /api/ {
        proxy_pass http://backend:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # SSE 流式响应支持
        proxy_buffering off;
        proxy_cache off;
        proxy_read_timeout 300s;
    }

    # WebSocket 升级（AI 对话实时推送）
    location /ws/ {
        proxy_pass http://backend:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_read_timeout 86400s;
    }

    # 请求体大小限制
    client_max_body_size 10M;
}
```

## 五、CI/CD 流水线

### 5.1 流水线总览

```
push to main ──→ lint + test ──→ build image ──→ push to ACR ──→ deploy to staging
                                                                       │
tag v*.*.* ────→ lint + test ──→ build image ──→ push to ACR ──→ deploy to production
                                                                  (手动确认)
```

### 5.2 镜像命名规范

```
registry.cn-hangzhou.aliyuncs.com/enterprise-workstation/backend:v{version}-{commit-sha}

# 示例
registry.cn-hangzhou.aliyuncs.com/enterprise-workstation/backend:v1.0.0-a3b4c5d
registry.cn-hangzhou.aliyuncs.com/enterprise-workstation/backend:latest
```

### 5.3 GitHub Actions Workflow

```yaml
# .github/workflows/deploy.yml

name: Build & Deploy

on:
  push:
    branches: [main]
    tags: ["v*.*.*"]

env:
  REGISTRY: registry.cn-hangzhou.aliyuncs.com
  IMAGE_NAME: enterprise-workstation/backend

jobs:
  # ─── 阶段 1：代码质量 ───
  lint-and-test:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:15-alpine
        env:
          POSTGRES_DB: test_db
          POSTGRES_USER: postgres
          POSTGRES_PASSWORD: postgres
        ports: ["5432:5432"]
        options: --health-cmd pg_isready --health-interval 10s --health-timeout 5s --health-retries 5
      redis:
        image: redis:7-alpine
        ports: ["6379:6379"]
        options: --health-cmd "redis-cli ping" --health-interval 10s --health-timeout 5s --health-retries 5
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v2
        with:
          version: 9
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: pnpm
      - run: pnpm install --frozen-lockfile
      - run: pnpm lint
      - run: pnpm prisma generate
      - run: pnpm test
        env:
          DATABASE_URL: postgresql://postgres:postgres@localhost:5432/test_db
          REDIS_URL: redis://localhost:6379/0
          NODE_ENV: test

  # ─── 阶段 2：构建并推送镜像 ───
  build-and-push:
    needs: lint-and-test
    runs-on: ubuntu-latest
    outputs:
      image_tag: ${{ steps.meta.outputs.tags }}
    steps:
      - uses: actions/checkout@v4

      - name: Login to ACR
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ secrets.ACR_USERNAME }}
          password: ${{ secrets.ACR_PASSWORD }}

      - name: Extract metadata
        id: meta
        run: |
          SHA=$(git rev-parse --short HEAD)
          if [[ "$GITHUB_REF" == refs/tags/v* ]]; then
            VERSION=${GITHUB_REF#refs/tags/}
            echo "tags=${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${VERSION}-${SHA}" >> $GITHUB_OUTPUT
          else
            echo "tags=${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:staging-${SHA}" >> $GITHUB_OUTPUT
          fi

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          target: production

  # ─── 阶段 3：部署到 staging ───
  deploy-staging:
    needs: build-and-push
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to staging
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.STAGING_HOST }}
          username: ${{ secrets.STAGING_USER }}
          key: ${{ secrets.STAGING_SSH_KEY }}
          script: |
            cd /opt/enterprise-workstation
            export IMAGE_TAG=${{ needs.build-and-push.outputs.image_tag }}
            docker compose -f docker-compose.yml -f docker-compose.prod.yml pull
            docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d --remove-orphans
            docker image prune -f

  # ─── 阶段 4：部署到 production（仅 tag 触发，需手动确认） ───
  deploy-production:
    needs: build-and-push
    if: startsWith(github.ref, 'refs/tags/v')
    runs-on: ubuntu-latest
    environment: production  # 需要在 GitHub 设置 environment protection rules
    steps:
      - name: Deploy to production
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.PRODUCTION_HOST }}
          username: ${{ secrets.PRODUCTION_USER }}
          key: ${{ secrets.PRODUCTION_SSH_KEY }}
          script: |
            cd /opt/enterprise-workstation
            export IMAGE_TAG=${{ needs.build-and-push.outputs.image_tag }}
            docker compose -f docker-compose.yml -f docker-compose.prod.yml pull
            docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d --remove-orphans
            docker image prune -f
```

### 6.4 环境变量模板 `.env.example`

# ═══════════════════════════════════════════

## 七、域名与证书规划

### 7.1 域名规划

| 域名 | 用途 | 指向 |
|------|------|------|
| `api.cebrenis.com` | API 服务入口 | ECS 公网 IP（Nginx） |
| `app.cebrenis.com` | Web 端（H5） | ECS 公网 IP 或 OSS 静态托管 |
| `cebrenis.com` | 官网（后期） | 同上 |

### 7.2 小程序域名白名单

微信小程序需要在后台配置以下域名白名单：

| 类型 | 域名 |
|------|------|
| request 合法域名 | `https://api.cebrenis.com` |
| socket 合法域名 | `wss://api.cebrenis.com` |
| uploadFile 合法域名 | `https://api.cebrenis.com` |
| downloadFile 合法域名 | `https://api.cebrenis.com` |

> 所有域名必须为 HTTPS，且已完成 ICP 备案。

### 7.3 SSL 证书方案

| 方案 | 适用场景 | 费用 | 自动续期 |
|------|---------|------|---------|
| 阿里云免费证书 | Phase 1 初期 | ¥0（每年 20 张免费） | 需手动续期 |
| Let's Encrypt + certbot | 推荐长期方案 | ¥0 | 支持自动续期 |
| 阿里云付费通配符证书 | 多子域名场景 | ¥1000+/年 | 自动续期 |

**推荐方案**：Phase 1 使用 Let's Encrypt + certbot 自动续期：

```bash
# 安装 certbot
apt install certbot python3-certbot-nginx

# 申请证书
certbot --nginx -d api.cebrenis.com -d app.cebrenis.com

# 自动续期（certbot 自动添加 cron）
certbot renew --dry-run
```

## 八、备份与恢复

> 本节补充主文档（第十四节）的具体 OSS 配置和 Docker 集成方案。备份目标（RPO < 1h、RTO < 4h）和备份策略详见 [01-主文档 · 第十四节](01-企业AI工作站-主文档.md#十四数据备份与灾备)。

### 8.1 OSS Bucket 规划

| Bucket 名称 | 存储类型 | 用途 |
|-------------|---------|------|
| `ew-backup-prod` | 标准存储 | 生产备份（WAL + 每日全量） |
| `ew-backup-staging` | 标准存储 | 预发布备份 |
| `ew-files-prod` | 标准存储 | 业务文件（报表导出等） |

### 8.2 Bucket 目录结构

```
ew-backup-prod/
├── wal/                          # WAL 归档（每小时）
│   └── 000000010000000000000001
├── daily/                        # 每日全量备份
│   ├── 20260329.dump
│   └── 20260330.dump
├── monthly/                      # 月度归档（低频存储）
│   └── 202603.dump
└── redis/                        # Redis RDB 快照
    └── 20260329-dump.rdb
```

### 8.3 生命周期规则

| 目录 | 规则 | 说明 |
|------|------|------|
| `wal/` | 7 天后删除 | WAL 保留 7 天 |
| `daily/` | 30 天后转低频，90 天后删除 | 每日备份保留 30 天 |
| `monthly/` | 365 天后转归档存储 | 月度备份保留 1 年 |
| `redis/` | 7 天后删除 | Redis 快照保留 7 天 |

### 8.4 备份脚本 Docker 集成

```yaml
# docker-compose.prod.yml 追加备份服务

services:
  backup:
    image: postgres:15-alpine
    env_file:
      - .env.production
    volumes:
      - ./scripts/backup.sh:/usr/local/bin/backup.sh:ro
      - backup_tmp:/tmp/backup
    entrypoint: ["crond", "-f", "-d", "8"]
    restart: always
```

```bash
#!/bin/bash
# scripts/backup.sh — 每日全量备份脚本

set -euo pipefail

DATE=$(date +%Y%m%d)
DUMP_FILE="/tmp/backup/${DATE}.dump"

# 1. 导出数据库
pg_dump -Fc -h "$DB_HOST" -U "$DB_USER" -d enterprise_workstation > "$DUMP_FILE"

# 2. 上传到 OSS
ossutil cp "$DUMP_FILE" "oss://ew-backup-prod/daily/${DATE}.dump" \
  --access-key-id "$OSS_ACCESS_KEY_ID" \
  --access-key-secret "$OSS_ACCESS_KEY_SECRET" \
  --endpoint "$OSS_ENDPOINT"

# 3. 清理本地临时文件
rm -f "$DUMP_FILE"

# 4. 上传 Redis 快照
redis-cli -h "$REDIS_HOST" -a "$REDIS_PASSWORD" BGSAVE
sleep 5
ossutil cp /data/dump.rdb "oss://ew-backup-prod/redis/${DATE}-dump.rdb" \
  --access-key-id "$OSS_ACCESS_KEY_ID" \
  --access-key-secret "$OSS_ACCESS_KEY_SECRET" \
  --endpoint "$OSS_ENDPOINT"

echo "[$(date)] Backup completed: ${DATE}"
```

### 8.5 恢复流程

```bash
# 1. 从 OSS 下载备份
ossutil cp oss://ew-backup-prod/daily/20260329.dump /tmp/restore.dump

# 2. 恢复到目标数据库
pg_restore -h $DB_HOST -U $DB_USER -d enterprise_workstation -c /tmp/restore.dump

# 3. 验证数据完整性
psql -h $DB_HOST -U $DB_USER -d enterprise_workstation -c "SELECT count(*) FROM tenants;"
```
