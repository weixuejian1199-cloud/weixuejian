## 一、核心设计哲学

### 第一原则：对话即操作，没有后台

```
传统企业软件：
  员工 → 登录后台 → 找菜单 → 填表单 → 提交 → 等审批 → 看结果
  培训成本高，操作路径长，每个功能都是新的学习曲线

我们的产品：
  员工 → 打开小程序 → 跟AI说话 → AI理解+确认+执行 → 完成
  零培训，零表单，零后台编辑权限
```

**这不是一个"带AI功能的管理系统"，是一个"用对话替代所有操作的AI操作系统"。**

### 真实场景说明：AIBI小酮营养师工具

运营部门的小张负责AIBI小酮（生酮社群AI营养师）。她的所有操作：

```
场景1：更新知识库
  ┌──────────────────────────────────────────────┐
  │  小张：我们新出了一款椰子油MCT粉，             │
  │  成分是C8+C10中链脂肪酸，每份5g净碳水，        │
  │  适合生酮期间冲咖啡或奶昔用                     │
  │                                                │
  │  小酮：收到！我整理成知识条目：                  │
  │  📋 产品：椰子油MCT粉                          │
  │  📋 成分：C8+C10中链脂肪酸                     │
  │  📋 净碳水：5g/份                              │
  │  📋 用法：冲咖啡/奶昔                          │
  │  确认添加到知识库吗？                           │
  │                                                │
  │  小张：确认                                     │
  │                                                │
  │  小酮：已添加 ✓ 现在有人问MCT粉我就能回答了     │
  └──────────────────────────────────────────────┘

场景2：调整回复风格
  ┌──────────────────────────────────────────────┐
  │  小张：回复太长了，控制在50字以内，              │
  │  语气再亲切一点，少用专业术语                    │
  │                                                │
  │  小酮：明白！调整内容：                         │
  │  1. 回复长度上限：100字 → 50字                  │
  │  2. 语气：专业严谨 → 亲切口语化                 │
  │  3. 术语处理：用大白话替代专业词                 │
  │  确认调整吗？                                   │
  │                                                │
  │  小张：确认                                     │
  │                                                │
  │  小酮：已更新 ✓                                 │
  └──────────────────────────────────────────────┘

场景3：测试效果
  ┌──────────────────────────────────────────────┐
  │  小张：模拟一下，用户问"刚开始生酮头疼正常吗"   │
  │                                                │
  │  小酮：📝 测试模式（不影响线上）                │
  │                                                │
  │  模拟回复：                                     │
  │  「亲，这是正常的酮流感哦～多喝水，              │
  │   加点盐就好啦，一般3-5天消失！」               │
  │                                                │
  │  字数：32字 ✓                                   │
  │  风格：亲切口语 ✓                               │
  │  满意这个效果吗？                               │
  │                                                │
  │  小张：可以，就这样                             │
  └──────────────────────────────────────────────┘
```

**注意以下事实：**
- 小张全程没有打开任何"知识库管理页面"
- 小张全程没有填写任何表单
- 小张全程没有编辑任何文档
- 小张甚至不知道"知识库""配置""参数"这些概念
- 她只是在跟小酮聊天，小酮帮她完成了一切

**这就是我们的产品。**

### 设计原则总结

| 原则 | 说明 | 反例（不做） |
|------|------|-------------|
| **对话即一切** | 所有操作通过自然语言完成 | 不做表单页、不做配置面板、不做后台编辑器 |
| **确认即生效** | Agent 理解→展示变更→用户确认→立即生效 | 不做"保存"按钮、不做"提交审批"流程 |
| **测试即所见** | 用户说"试一下"，Agent 当场模拟效果 | 不做"预览环境"、不做"沙盒模式切换" |
| **极简信息** | 每屏只展示当前最重要的信息 | 不做信息轰炸、不做多层嵌套导航 |
| **角色即边界** | 不同角色看到不同的Tab和能力 | 不做"隐藏菜单"、不做"灰色不可点击" |

## 三、色彩系统

### 3.1 基础色板

```
背景色（暖色调 — 参考扣子的米灰质感，温暖不刺眼，区别于冷灰系产品）
├── 页面背景    #F8F7F4    暖米灰（主背景，有温度感）
├── 卡片背景    #FFFFFF    纯白（卡片、对话区，与暖底形成层次）
├── 输入区背景  #F3F2EF    暖浅灰（输入框、搜索框底色）
├── 侧栏背景    #F5F4F1    暖灰（Web端左侧栏、对话历史区，参考扣子侧栏）
└── 分隔线      #E8E6E1    暖淡灰（极细分隔）

主色（AI 智能感 — 保持蓝紫，不跟随暖色，形成品牌辨识度）
├── 主色        #4F6EF7    蓝（按钮、链接、主操作）
├── 主色渐变    #4F6EF7 → #7C5CFC    蓝→紫（AI气泡、特殊强调）
├── 主色浅底    #EEF1FE    极浅蓝（选中态、hover态）
└── 主色深      #3451D1    深蓝（按钮pressed态）

文字色（微带暖调，与背景协调）
├── 主文字      #1D1D1F    近黑（标题、正文）
├── 次文字      #4E4E52    深灰（说明文字、副标题）
├── 辅助文字    #8E8E93    中灰（时间戳、提示）
└── 占位文字    #C7C7CC    浅灰（placeholder）

语义色（数据场景必需）
├── 成功/增长   #00B42A    绿（正增长、已完成、健康）
├── 警告/注意   #FF7D00    橙（库存预警、配额接近上限）
├── 错误/下降   #F53F3F    红（负增长、异常、错误）
├── 信息/中性   #3491FA    蓝（通知、提示）
└── 已处理      #86909C    灰（已读、已处理、过期）
```

### 3.2 数据可视化色板

```
多系列图表（店铺/平台对比时使用，最多6色）
├── 系列1    #4F6EF7    蓝（默认第一系列）
├── 系列2    #7C5CFC    紫
├── 系列3    #14C9C9    青
├── 系列4    #F7BA1E    金
├── 系列5    #F77234    橙
└── 系列6    #9FDB1D    黄绿

涨跌配色（财务场景）
├── 涨/盈利    #00B42A    绿色字 + #E8FFEA 浅绿底
├── 跌/亏损    #F53F3F    红色字 + #FFECE8 浅红底
└── 持平       #86909C    灰色字
```

### 3.3 暗色模式（Phase 2，非首发）

首发版本仅提供浅色模式。暗色模式作为 Phase 2 可选项，不是默认。

## 四、字体与排版

### 4.1 字体栈

```css
/* 正文（小程序+Web通用） */
font-family: -apple-system, BlinkMacSystemFont, 'PingFang SC',
             'Helvetica Neue', 'Microsoft YaHei', sans-serif;
```

### 4.2 字号规范

| 用途 | 字号 | 字重 | 行高 | 场景 |
|------|------|------|------|------|
| 大标题 | 20px | Bold (600) | 28px | 页面标题 |
| 标题 | 17px | Medium (500) | 24px | 卡片标题、对话角色名 |
| 正文 | 15px | Regular (400) | 22px | 对话内容、列表项 |
| 辅助 | 13px | Regular (400) | 18px | 时间戳、标签、说明 |
| 小字 | 11px | Regular (400) | 16px | 角标、极小提示 |

| 金额正文 | 17px | Medium (500) | 24px | 表格中的金额 |

### 4.3 数字格式

| 类型 | 格式 | 示例 |
|------|------|------|
| 金额 | 千位分隔符 + 两位小数 | ¥328,400.00 |

| 日期 | 短格式 | 03-29 / 2026-03-29 |

### 5.2 核心组件：AIChatBox

**这是整个产品最重要的组件。** 只有一个，复用到所有场景。

```
┌─────────────────────────────────────────┐
```

#### AIChatBox 子组件详细规格

AIChatBox 由 7 个子组件组成，以下为完整的 Props 和交互规格：

**1. MessageBubble（消息气泡）**

```typescript
interface MessageBubbleProps {
  content: string;                        // 消息内容（纯文本或Markdown）
  sender: 'user' | 'ai';                 // 发送方
  timestamp: number;                      // 时间戳
  status: 'sending' | 'sent' | 'error';  // 发送状态
  onRetry?: () => void;                   // 重试回调（仅error态）
  onCopy?: () => void;                    // 复制回调
  onRegenerate?: () => void;              // 重新生成回调（仅AI消息）
}
```

交互规则：
- 用户消息：右对齐，蓝紫渐变底（`--color-brand-gradient-start` → `--color-brand-gradient-end`），白色文字
- AI 消息：左对齐，白底（`--color-bg-secondary`），主文字色
- 长按操作：用户消息→复制；AI消息→复制 / 重新生成
- 发送失败：消息右下角显示红色感叹号（`--color-error`），点击触发 `onRetry`
- 发送中：消息右下角显示灰色旋转图标

**2. StreamingText（流式文本渲染器）**

```typescript
interface StreamingTextProps {
  text: string;                           // 实时更新的文本内容
  isStreaming: boolean;                   // 是否正在流式输出
  onStop?: () => void;                    // 停止生成回调
}
```

交互规则：
- Markdown 实时渲染：支持加粗、列表、代码块、表格
- 流式输出时底部显示光标闪烁动画（CSS animation，非 JS 定时器）
- 正在输出时底部浮现「停止生成」按钮，点击调用 `onStop`
- 输出完成后（`isStreaming=false`）光标消失，显示「复制 | 重新生成」
- 代码块渲染：深色背景（`#1E1E1E`）+ 语法高亮 + 右上角复制按钮

**3. ConfirmationCard（确认卡片）**

```typescript
interface ConfirmationCardProps {
  title: string;                          // 卡片标题
  fields: Array<{                         // 变更字段列表
    label: string;
    oldValue?: string;
    newValue: string;
  }>;
  riskLevel: 'low' | 'medium' | 'high';  // 风险等级
  expiresAt?: number;                     // 过期时间戳（高风险必填）
  confirmToken: string;                   // 安全Token
  onConfirm: (token: string, reason?: string) => void;
  onCancel: () => void;
}
```

交互规则：
- 低风险（`riskLevel='low'`）：绿色左边框（`--color-success`），直接点确认
- 中风险（`riskLevel='medium'`）：橙色左边框（`--color-warning`），需点击确认
- 高风险（`riskLevel='high'`）：红色左边框（`--color-error`），显示倒计时（5分钟），需输入理由才能确认
- 过期后：整体灰色（`--color-neutral`），按钮不可点击，显示「已过期，请重新发起」
- 已确认/已取消：卡片变灰色不可操作态，显示操作结果标签

**4. ProcessIndicator（过程指示器）**

```typescript
interface ProcessIndicatorProps {
  steps: Array<{
    name: string;                         // 步骤名称
    detail?: string;                      // 步骤详情/结果摘要
  }>;
  currentStep: number;                    // 当前执行到的步骤索引
  totalTime?: number;                     // 总耗时（完成后显示）
  collapsed?: boolean;                    // 是否折叠态
  onToggle?: () => void;                  // 切换折叠
}
```

交互规则：
- 垂直步骤条：左侧 2px 进度竖线
- 已完成步骤：绿色勾（`--color-success`）+ 正文色文字 + 可展开详情
- 当前步骤：蓝色旋转图标（`--color-brand-primary`）+ 呼吸动效
- 未来步骤：灰色（`--color-text-placeholder`）+ 淡字
- 全部完成后自动折叠为一行「✅ N步完成（Xs）」，点击可展开

**5. DataCard（数据卡片）**

```typescript
interface DataCardProps {
  title: string;                          // 卡片标题
  value: string | number;                 // 核心数字
  unit?: string;                          // 单位（元/单/件）
  trend: 'up' | 'down' | 'flat';         // 趋势方向
  trendValue: string;                     // 趋势百分比（如"+12.3%"）
  chartData?: number[];                   // 可选 sparkline 数据点
  subMetrics?: Array<{                    // 子指标（最多3个）
    label: string;
    value: string;
  }>;
  onClick?: () => void;                   // 点击进入详情
}
```

交互规则：
- 主指标：`--font-size-2xl` + `--font-weight-bold` + 等宽数字字体
- 趋势箭头：up=绿色↑（`--color-success`），down=红色↓（`--color-error`），flat=灰色→
- sparkline 迷你图：高度 32px，无坐标轴，仅显示趋势线条
- 子指标：三等分横排，`--font-size-sm` + `--color-text-tertiary`
- 点击整个卡片：`scale(0.98)` 按压反馈 → 触发 `onClick`

**6. QuickActions（快捷操作栏）**

```typescript
interface QuickActionsProps {
  actions: Array<{
    icon: string;                         // 图标
    label: string;                        // 两行标签
    message: string;                      // 预填消息内容
  }>;
  userRole: string;                       // 当前用户角色（动态过滤）
  onAction: (message: string) => void;    // 预填+自动发送

```

权限-能力映射表：

| 角色 | 可见快捷能力 |
|------|-------------|
| super_admin | 全部（今日销售汇总/利润报表/库存预警/全局分析/成本分析/对账/退货审核...） |
| admin | 全部（同 super_admin，不含系统驾驶舱入口） |
| finance | 本月流水/报税数据/平台手续费/成本分析/供应商对账（Phase 2+）/利润报表 |
| operation | 今日本店数据/投流ROI/热销排行/文案生成/库存预警/竞品分析 |
| cs（客服） | 退货审核/工单处理/FAQ查询/客诉统计/话术推荐 |
| purchase（采购） | 待结算供应商/退款归因/生成结算单/供应商比价 |

交互规则：
- 横向滚动，不换行
- 仅在对话首页（无历史消息时）显示
- 点击行为：预填对话消息 + 自动发送（调用 `onAction`）

**7. EscalateNotice（转人工通知）**

```typescript
interface EscalateNoticeProps {
```

交互规则：
- 黄色提示条：背景 `--color-warning-bg` + 左边框 `--color-warning`

- 底部链接：「查看工单 #{ticketId}」可点击跳转工单卡片
- 有指派人时显示：「已指派给 {assignee}」

**风险等级与 UI 映射：**

| 风险等级 | 标识 | 确认按钮色 | 有效期 | 场景 |
|---------|------|----------|--------|------|
| 高风险（金额/不可逆） | 红色标签 | 红色 | 5分钟倒计时 | 结算确认、大额退款、删除、部署 |
| 中风险（配置变更） | 橙色标签 | 主色蓝 | 无时限 | 工具配置、权限变更、价格调整 |
| 低风险（信息确认） | 无标签 | 主色蓝 | 无时限 | 知识库添加、报表生成 |

**技术实现：**
- 高风险操作使用 `confirmToken` 机制（见 Agent 编排文档第八节），卡片展示倒计时
- 确认卡片由 AI 响应中的结构化数据渲染，不是前端硬编码
- AI 返回格式：`{ type: 'confirmation_card', risk: 'high', fields: [...], confirmToken: '...' }`
- 前端 AIChatBox 组件识别 `confirmation_card` 类型，渲染对应卡片组件

**confirmToken 安全机制：**

```
生成规则：
├── 格式：UUID v4
├── 绑定：userId + tenantId + 操作类型 + 风险等级
├── 生成时机：AI Agent 判定需要确认时，由后端生成

存储与校验：
├── 存储位置：Redis Hash
│   key:   confirm:{token}
│   field:  userId, tenantId, operationType, riskLevel, payload, createdAt
│   TTL:    高风险=5分钟 / 中风险=30分钟 / 低风险=60分钟
├── 校验流程：
│   1. 前端发送 POST /api/confirm/{token}
│   2. 后端从 Redis 取出 Token 数据
│   3. 校验三重匹配：token + userId + tenantId
│   4. 校验 TTL 未过期
│   5. 校验操作类型与当前请求一致
│   6. 通过 → 执行操作 + 立即删除 Token
│   7. 失败 → 返回具体错误（token_expired / token_mismatch / token_used）

防重放机制：
├── 单次使用：确认执行成功后立即从 Redis 删除
├── 前端防抖：确认按钮点击后立即 disable + 显示 loading
├── 后端幂等：执行前先 Redis WATCH + MULTI 原子操作
└── 审计日志：每次确认操作记录到 audit_log 表（userId/action/token/timestamp）
```

## 十二、动效与微交互

### 12.1 核心动效（克制使用，小程序性能优先）

```
页面切换：原生栈式动画（左右滑入滑出）
卡片点击：scale(0.98) 0.1s（轻微按压反馈）
按钮点击：opacity 0.7 0.1s（按下变暗）
下拉刷新：原生动画 + 主色转圈
AI 流式输出：逐字渲染（不用打字机光标，省性能）
确认成功：✓ 图标 + 轻微弹跳 0.3s
消息发送：消息从下方滑入 0.2s ease-out
```

### 12.2 加载态

```
页面加载：骨架屏（灰色占位块模拟内容布局）
列表加载：底部三个跳动圆点
AI 思考中：「正在思考...」+ 三个渐变呼吸点
工具调用中：「🔍 正在查询{工具名}...」+ 进度条
图表加载：先渲染坐标轴 → 数据点依次出现
```

## 十三、我们和竞品的定位差异

**我们参考了扣子和悟空，但我们不是它们中的任何一个。**

| 维度 | 扣子（Coze） | 悟空（DingTalk） | 我们 |
|------|-------------|-----------------|------|
| **定位** | AI 协作平台 + Agent 搭建 | 企业 AI 执行工作台 | **企业 AI 操作系统（对话即一切）** |
| **用户** | 开发者/创作者搭建Agent | 企业员工执行任务 | **电商企业全角色（老板到客服）** |
| **核心界面** | 对话 + 工作流编辑器 + 代码 | 任务输入 + 技能卡片 | **纯对话。没有编辑器，没有后台** |
| **配置方式** | 后台表单 + 可视化编排 | 后台技能配置 | **对话中完成。员工说"调一下"，Agent确认后生效** |
| **数据分析** | 慢（调Python终端，几分钟） | 未知 | **秒级（本地分析引擎 + 本地DB直查）** |
| **过程展示** | 完整代码终端（员工看不懂） | 简洁 | **业务步骤指示器（员工看得懂）** |
| **工具市场** | Bot 商店（可搭建） | 技能中心 | **工具=Agent，点进去就是对话，不是配置页** |
| **学我什么** | 暖色调、对话布局、过程可视化 | 极简侧栏、任务导向语气、技能卡片 | — |
| **不学什么** | 代码编辑器、工作流画布、积分体系 | 云电脑操控、DingTalk深度耦合 | — |

**一句话：扣子让你搭Agent，悟空让Agent帮你操作电脑，我们让你跟Agent说句话就把事办了。**

## 十四、关键决策速查

| 问题 | 答案 |
|------|------|
| 员工能看到后台吗？ | **不能。没有后台。一切通过对话完成。** |
| 工具怎么配置？ | 跟工具 Agent 对话，Agent 确认后生效 |
| 知识库怎么管理？ | 跟工具 Agent 对话，告诉它新知识，它自动整理写入 |
| 需要培训员工吗？ | 不需要。会说话就会用。Agent 欢迎语就是引导。 |
| 暗色模式做吗？ | Phase 2 可选，不是默认 |
| 首页放什么？ | 待办+核心指标+AI洞察。任务驱动，不是信息轰炸。 |
| 前端核心组件是什么？ | AIChatBox。一个组件，传不同 agent 参数，复用所有场景。 |
| 设计参考谁？ | 工作台=悟空，对话=扣子+Gemini，数据=Stripe |
| Design Token 怎么用？ | 所有样式引用 CSS Custom Properties，不硬编码色值/字号 |
| 图表用什么库？ | echarts（按需引入 Line/Bar/Pie），Phase 1 三种够用 |
| 无障碍做到什么标准？ | WCAG AA，对比度4.5:1，ARIA标签，键盘导航 |
| 分包怎么分？ | 主包(框架+首页+AIChatBox) + 4个分包(对话/驾驶舱/工具/商城) |
| 确认卡片安全机制？ | confirmToken(UUID v4) + Redis TTL + userId+tenantId三重校验 + 单次使用 |
| 哪些页面必须独立存在？ | 7个（登录/首页/对话/历史/工具市场/设置/驾驶舱），其余嵌入对话 |
