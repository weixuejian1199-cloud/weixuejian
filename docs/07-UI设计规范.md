## 一、核心设计哲学

### 第一原则：对话即操作，没有后台

```
传统企业软件：
  员工 → 登录后台 → 找菜单 → 填表单 → 提交 → 等审批 → 看结果
  培训成本高，操作路径长，每个功能都是新的学习曲线

我们的产品：
  员工 → 打开小程序 → 跟AI说话 → AI理解+确认+执行 → 完成
  零培训，零表单，零后台编辑权限
```

**这不是一个"带AI功能的管理系统"，是一个"用对话替代所有操作的AI操作系统"。**

### 真实场景说明：AIBI小酮营养师工具

运营部门的小张负责AIBI小酮（生酮社群AI营养师）。她的所有操作：

```
场景1：更新知识库
  ┌──────────────────────────────────────────────┐
  │  小张：我们新出了一款椰子油MCT粉，             │
  │  成分是C8+C10中链脂肪酸，每份5g净碳水，        │
  │  适合生酮期间冲咖啡或奶昔用                     │
  │                                                │
  │  小酮：收到！我整理成知识条目：                  │
  │  📋 产品：椰子油MCT粉                          │
  │  📋 成分：C8+C10中链脂肪酸                     │
  │  📋 净碳水：5g/份                              │
  │  📋 用法：冲咖啡/奶昔                          │
  │  确认添加到知识库吗？                           │
  │                                                │
  │  小张：确认                                     │
  │                                                │
  │  小酮：已添加 ✓ 现在有人问MCT粉我就能回答了     │
  └──────────────────────────────────────────────┘

场景2：调整回复风格
  ┌──────────────────────────────────────────────┐
  │  小张：回复太长了，控制在50字以内，              │
  │  语气再亲切一点，少用专业术语                    │
  │                                                │
  │  小酮：明白！调整内容：                         │
  │  1. 回复长度上限：100字 → 50字                  │
  │  2. 语气：专业严谨 → 亲切口语化                 │
  │  3. 术语处理：用大白话替代专业词                 │
  │  确认调整吗？                                   │
  │                                                │
  │  小张：确认                                     │
  │                                                │
  │  小酮：已更新 ✓                                 │
  └──────────────────────────────────────────────┘

场景3：测试效果
  ┌──────────────────────────────────────────────┐
  │  小张：模拟一下，用户问"刚开始生酮头疼正常吗"   │
  │                                                │
  │  小酮：📝 测试模式（不影响线上）                │
  │                                                │
  │  模拟回复：                                     │
  │  「亲，这是正常的酮流感哦～多喝水，              │
  │   加点盐就好啦，一般3-5天消失！」               │
  │                                                │
  │  字数：32字 ✓                                   │
  │  风格：亲切口语 ✓                               │
  │  满意这个效果吗？                               │
  │                                                │
  │  小张：可以，就这样                             │
  └──────────────────────────────────────────────┘
```

**注意以下事实：**
- 小张全程没有打开任何"知识库管理页面"
- 小张全程没有填写任何表单
- 小张全程没有编辑任何文档
- 小张甚至不知道"知识库""配置""参数"这些概念
- 她只是在跟小酮聊天，小酮帮她完成了一切

**这就是我们的产品。**

### 设计原则总结

| 原则 | 说明 | 反例（不做） |
|------|------|-------------|
| **对话即一切** | 所有操作通过自然语言完成 | 不做表单页、不做配置面板、不做后台编辑器 |
| **确认即生效** | Agent 理解→展示变更→用户确认→立即生效 | 不做"保存"按钮、不做"提交审批"流程 |
| **测试即所见** | 用户说"试一下"，Agent 当场模拟效果 | 不做"预览环境"、不做"沙盒模式切换" |
| **极简信息** | 每屏只展示当前最重要的信息 | 不做信息轰炸、不做多层嵌套导航 |
| **角色即边界** | 不同角色看到不同的Tab和能力 | 不做"隐藏菜单"、不做"灰色不可点击" |

## 三、色彩系统

### 3.1 基础色板

```
背景色（暖色调 — 参考扣子的米灰质感，温暖不刺眼，区别于冷灰系产品）
├── 页面背景    #F8F7F4    暖米灰（主背景，有温度感）
├── 卡片背景    #FFFFFF    纯白（卡片、对话区，与暖底形成层次）
├── 输入区背景  #F3F2EF    暖浅灰（输入框、搜索框底色）
├── 侧栏背景    #F5F4F1    暖灰（Web端左侧栏、对话历史区，参考扣子侧栏）
└── 分隔线      #E8E6E1    暖淡灰（极细分隔）

主色（AI 智能感 — 保持蓝紫，不跟随暖色，形成品牌辨识度）
├── 主色        #4F6EF7    蓝（按钮、链接、主操作）
├── 主色渐变    #4F6EF7 → #7C5CFC    蓝→紫（AI气泡、特殊强调）
├── 主色浅底    #EEF1FE    极浅蓝（选中态、hover态）
└── 主色深      #3451D1    深蓝（按钮pressed态）

文字色（微带暖调，与背景协调）
├── 主文字      #1D1D1F    近黑（标题、正文）
├── 次文字      #4E4E52    深灰（说明文字、副标题）
├── 辅助文字    #8E8E93    中灰（时间戳、提示）
└── 占位文字    #C7C7CC    浅灰（placeholder）

语义色（数据场景必需）
├── 成功/增长   #00B42A    绿（正增长、已完成、健康）
├── 警告/注意   #FF7D00    橙（库存预警、配额接近上限）
├── 错误/下降   #F53F3F    红（负增长、异常、错误）
├── 信息/中性   #3491FA    蓝（通知、提示）
└── 已处理      #86909C    灰（已读、已处理、过期）
```

### 3.2 数据可视化色板

## 四、字体与排版

### 4.1 字体栈

```css
/* 正文（小程序+Web通用） */
font-family: -apple-system, BlinkMacSystemFont, 'PingFang SC',
             'Helvetica Neue', 'Microsoft YaHei', sans-serif;

/* 数字/金额（等宽，提升数据可读性） */
font-family: 'DIN Alternate', 'Roboto Mono', 'SF Mono', monospace;
```

### 4.2 字号规范

| 用途 | 字号 | 字重 | 行高 | 场景 |
|------|------|------|------|------|
| 大标题 | 20px | Bold (600) | 28px | 页面标题 |
| 标题 | 17px | Medium (500) | 24px | 卡片标题、对话角色名 |
| 正文 | 15px | Regular (400) | 22px | 对话内容、列表项 |
| 辅助 | 13px | Regular (400) | 18px | 时间戳、标签、说明 |
| 小字 | 11px | Regular (400) | 16px | 角标、极小提示 |
| 金额大数 | 28px | Bold (600) | 36px | 卡片主指标（¥328,400） |
| 金额正文 | 17px | Medium (500) | 24px | 表格中的金额 |

### 4.3 数字格式

| 类型 | 格式 | 示例 |
|------|------|------|
| 金额 | 千位分隔符 + 两位小数 | ¥328,400.00 |
| 百分比 | 一位小数 + 涨跌符号 | +12.3% ↑ / -3.1% ↓ |
| 数量 | 千位分隔符 + 整数 | 1,256 单 |
| 日期 | 短格式 | 03-29 / 2026-03-29 |
| 时间 | 24小时制 | 14:30 |

### 5.3 核心组件：过程指示器（ProcessIndicator）

**设计背景**：扣子在执行复杂任务时会展示完整代码终端（Python import pandas...），过程可见但对员工无意义且非常慢。Qiyao分析引擎很快但没有过程。我们要两全：**展示业务步骤 + 极速完成**。

```
┌──────────────────────────────────────────┐
│  🤖 收到！正在帮你分析...                │
│                                          │
│  ┌──────────────────────────────────┐    │  ← 过程指示器（嵌入对话流）
│  │  ✅ 识别数据范围  抖店 · 03-22~28 │    │    白底卡片，左侧进度线
│  │  ✅ 查询订单数据  1,256 条         │    │    已完成=绿色✅
│  │  ✅ 计算核心指标  GMV/客单/转化率  │    │    进行中=蓝色🔄+呼吸动效
│  │  🔄 生成分析报告  ...              │    │    折叠态=一行"3步已完成"
│  └──────────────────────────────────┘    │
│                                          │
│  ──── 2-5秒后，结果直接出现 ────        │
│                                          │
│  📊 上周抖店销售分析                      │
│  ┌────────────────────────────────────┐  │
│  │  GMV ¥128,400    订单 1,256单       │  │
│  │  客单价 ¥102     转化率 3.2%        │  │
│  │  [趋势折线图]                       │  │
│  └────────────────────────────────────┘  │
│  较上上周 +8.3%，周三是峰值...            │
│  建议：周三的直播场次可以增加...           │
└──────────────────────────────────────────┘
```

**设计规范：**

```
过程指示器卡片
├── 背景：#FFFFFF 白底
├── 左侧：2px 进度竖线（已完成段 #00B42A 绿，进行中段 #4F6EF7 蓝）
├── 每步一行：图标 + 步骤名 + 简短结果
│   ├── ✅ 已完成步骤：#00B42A 绿色勾 + #4E4E52 正文
│   ├── 🔄 进行中步骤：#4F6EF7 蓝色旋转图标 + 呼吸动效
│   └── ⏳ 排队步骤：#C7C7CC 灰色 + 淡字
├── 圆角：8px
├── 内边距：12px
├── 步骤间距：8px
└── 完成后自动折叠为一行："✅ 4步分析完成（0.8s）" 可展开查看

与扣子的关键区别：
├── 我们展示的是「业务步骤」不是「代码执行」
│   扣子："$ python -c 'import pandas as pd...'"（员工看不懂）

```

**风险等级与 UI 映射：**

| 风险等级 | 标识 | 确认按钮色 | 有效期 | 场景 |
|---------|------|----------|--------|------|
| 高风险（金额/不可逆） | 红色标签 | 红色 | 5分钟倒计时 | 结算确认、大额退款、删除、部署 |
| 中风险（配置变更） | 橙色标签 | 主色蓝 | 无时限 | 工具配置、权限变更、价格调整 |
| 低风险（信息确认） | 无标签 | 主色蓝 | 无时限 | 知识库添加、报表生成 |

**技术实现：**
- 高风险操作使用 `confirmToken` 机制（见 Agent 编排文档第八节），卡片展示倒计时
- 确认卡片由 AI 响应中的结构化数据渲染，不是前端硬编码
- AI 返回格式：`{ type: 'confirmation_card', risk: 'high', fields: [...], confirmToken: '...' }`
- 前端 AIChatBox 组件识别 `confirmation_card` 类型，渲染对应卡片组件

## 七、交互规范

### 7.1 导航交互

```
底部 Tab 切换：
├── 无动画切换（小程序原生Tab限制）
├── 选中态：主色图标 + 主色文字
├── 未选中：#86909C 灰色
├── Tab 角标：红点（有新消息/待办时）
└── 各角色 Tab 配置见主文档第八节

页面内导航：
├── 返回：左上角箭头，统一样式
├── 下拉刷新：原生下拉 + 主色加载指示器
├── 上拉加载：底部 loading + "加载中..."
└── 空状态：插画 + 一句话说明 + 操作按钮
```

### 7.2 对话交互

```
消息发送：
├── 回车 / 点击发送按钮
├── 发送后输入框清空，消息立即出现在对话流
├── AI 开始流式回复（逐字输出）
├── 输出过程中可点击"停止生成"
└── 输出完成后消息底部显示操作项

工具调用中：
├── 显示"🔍 正在查询销售数据..."
├── 带呼吸动效的加载点
├── 结果返回后无缝插入对话流
└── 不跳转任何页面

确认交互（配置模式）：
├── Agent 展示变更确认卡片
├── 用户点"确认" → 立即生效 + 成功提示
├── 用户点"取消" → 不变更 + Agent 回复"好的，不做调整"
├── 确认卡片有5分钟有效期（超时自动失效）
└── 已确认/已取消的卡片变为灰色不可操作态

测试交互（测试模式）：
├── Agent 识别测试意图 → 顶部出现橙色"测试模式"标签
├── 模拟结果在橙色边框卡片中展示
├── 测试结果不影响任何线上数据和配置
├── 用户可连续测试多次
└── 用户说"满意/确认"→ 自动退出测试模式
```

### 7.3 数据交互

```
表格：
├── 横向可滑动（小程序屏幕窄）
├── 首列固定（通常是名称/平台）
├── 金额右对齐，等宽数字
├── 涨跌用颜色区分，不依赖图标（色盲友好考虑辅以箭头）
└── 点击行可展开详情（手风琴式）

图表：
├── 优先折线图（趋势）和柱状图（对比）
├── 颜色使用数据可视化色板
├── 不超过6个系列（超过用"其他"合并）
├── 点击数据点显示 tooltip
└── 图表下方提供"让AI分析"快捷入口
```

## 八、响应式适配

### 8.1 小程序端（Taro，主阵地）

```
设计基准：375px 宽度（iPhone 标准）
适配范围：320px - 428px

关键约束：
├── 底部 Tab 最多 5 个（微信限制）
├── 导航栏高度 44px + 状态栏（适配刘海屏）

```

├── < 768px：与小程序一致的移动端布局

├── 对话历史侧边栏（常驻左侧）

├── 拖拽上传文件

## 九、设计禁区（绝对不做）

| 禁止 | 原因 |
|------|------|
| ❌ 知识库管理后台页面 | 通过对话完成，不需要 |

| ❌ 图表首页大屏（花哨但无用） | 数据卡片精准展示，不做驾驶舱大屏 |

## 十、动效与微交互

### 10.1 核心动效（克制使用，小程序性能优先）

```
页面切换：原生栈式动画（左右滑入滑出）
卡片点击：scale(0.98) 0.1s（轻微按压反馈）
按钮点击：opacity 0.7 0.1s（按下变暗）
下拉刷新：原生动画 + 主色转圈
AI 流式输出：逐字渲染（不用打字机光标，省性能）
确认成功：✓ 图标 + 轻微弹跳 0.3s
消息发送：消息从下方滑入 0.2s ease-out
```

### 10.2 加载态

```
页面加载：骨架屏（灰色占位块模拟内容布局）
列表加载：底部三个跳动圆点
AI 思考中：「正在思考...」+ 三个渐变呼吸点
工具调用中：「🔍 正在查询{工具名}...」+ 进度条
图表加载：先渲染坐标轴 → 数据点依次出现
```

## 十一、我们和竞品的定位差异

**我们参考了扣子和悟空，但我们不是它们中的任何一个。**

| 维度 | 扣子（Coze） | 悟空（DingTalk） | 我们 |
|------|-------------|-----------------|------|
| **定位** | AI 协作平台 + Agent 搭建 | 企业 AI 执行工作台 | **企业 AI 操作系统（对话即一切）** |
| **用户** | 开发者/创作者搭建Agent | 企业员工执行任务 | **电商企业全角色（老板到客服）** |
| **核心界面** | 对话 + 工作流编辑器 + 代码 | 任务输入 + 技能卡片 | **纯对话。没有编辑器，没有后台** |
| **配置方式** | 后台表单 + 可视化编排 | 后台技能配置 | **对话中完成。员工说"调一下"，Agent确认后生效** |
| **数据分析** | 慢（调Python终端，几分钟） | 未知 | **秒级（Qiyao引擎 + 本地DB直查）** |
| **过程展示** | 完整代码终端（员工看不懂） | 简洁 | **业务步骤指示器（员工看得懂）** |
| **工具市场** | Bot 商店（可搭建） | 技能中心 | **工具=Agent，点进去就是对话，不是配置页** |
| **学我什么** | 暖色调、对话布局、过程可视化 | 极简侧栏、任务导向语气、技能卡片 | — |
| **不学什么** | 代码编辑器、工作流画布、积分体系 | 云电脑操控、DingTalk深度耦合 | — |

**一句话：扣子让你搭Agent，悟空让Agent帮你操作电脑，我们让你跟Agent说句话就把事办了。**

## 十二、关键决策速查

| 问题 | 答案 |
|------|------|
| 员工能看到后台吗？ | **不能。没有后台。一切通过对话完成。** |
| 工具怎么配置？ | 跟工具 Agent 对话，Agent 确认后生效 |
| 知识库怎么管理？ | 跟工具 Agent 对话，告诉它新知识，它自动整理写入 |
| 需要培训员工吗？ | 不需要。会说话就会用。Agent 欢迎语就是引导。 |
| 暗色模式做吗？ | Phase 2 可选，不是默认 |
| 首页放什么？ | 待办+核心指标+AI洞察。任务驱动，不是信息轰炸。 |
| 前端核心组件是什么？ | AIChatBox。一个组件，传不同 agent 参数，复用所有场景。 |
| 设计参考谁？ | 工作台=悟空，对话=扣子+Gemini，数据=Stripe |
