## 一、项目定位

**不是普通工作台，是企业 AI 操作系统。**

核心价值：把数据、AI、业务流程三合一，让每个岗位的员工只需要"问"，不需要"操作"。未来所有员工只需拿着小程序登录，就能完成公司所有相关业务。

---

## 二、核心业务场景

### 现有能力
- 抖店、天猫、美团闪购等平台数据导入
- 多维度数据报告（推广率、达人、订单、气泡柱图表）
- 手工数据接入

### 近期目标
- 接入聚水潭 API，实现各店铺实时数据分析
- 各岗位 AI 工作助手（自然语言查询数据）

### 长期愿景
- 管理层：语音/文字问任意业务数据，实时汇总
- 财务：替代金蝶用友，自动生成报税数据
- 运营：投流建议、活动策划、达人数据分析
- 采购：供应商自动结算对账（Phase 2+）
- 员工从一周上班6天 → 3-4天，效率翻倍

## 五、多租户架构

**这是 SaaS 产品，未来要卖给其他企业，多租户从第一天设计进去。**

核心原则：
- 所有数据表必须包含 `tenant_id` 字段
- 所有数据库查询必须带 `tenant_id` 过滤
- 不同租户数据物理上在同一数据库，逻辑完全隔离
- 通过域名或企业码识别租户（如 `company.saas.com`）

#### 支付状态机

完整状态流转：

- `pending`（待支付）：订单创建时生成支付单，此时 expiredAt = now + 30分钟
- `pending → processing`（支付中）：用户在客户端发起支付，后端调用渠道统一下单接口成功
- `processing → paid`（已支付）：收到渠道异步回调，验签通过，金额匹配，写入 paidAmount + channelTradeNo + paidAt
- `processing → failed`（支付失败）：渠道回调返回失败，或主动查询确认失败
- `pending → closed`（已关闭）：超过 expiredAt 未支付，BullMQ 定时任务扫描关单
- `paid → refunding`（退款中）：发起退款请求到渠道
- `refunding → refunded`（已退款）：收到渠道退款成功回调

#### 幂等性保证

- **下单幂等**：以 `paymentNo` 作为幂等键，同一 paymentNo 重复请求返回已有支付单，不重复创建
- **回调幂等**：处理回调前先检查 Payment.status，已为 `paid` 则直接返回 success 不重复处理
- **退款幂等**：以 `refundNo` 作为幂等键，渠道侧同一 refundNo 不会重复退款

#### 超时与重试

- 支付超时：30分钟未完成支付，BullMQ 延迟任务自动关单（更新 status → closed，释放库存）
- 回调丢失兜底：支付发起后 5 分钟，如果仍为 processing，BullMQ 触发主动查询渠道订单状态
- 退款超时：退款发起后 24 小时仍为 processing，标记异常 + 告警管理员人工处理

#### 微信/支付宝回调处理

```
POST /webhook/wxpay
  → 验证签名（微信V3 AEAD_AES_256_GCM 解密）
  → 提取 out_trade_no（即 paymentNo）+ transaction_id + 金额
  → 数据库事务：
    1. 查 Payment by paymentNo，检查 status
    2. 校验金额一致（分→元转换后与 amount 比对）
    3. 更新 Payment：status=paid, paidAmount, channelTradeNo, paidAt, callbackRaw
    4. 更新 MallOrder：status=paid, paidAt
    5. 写 AuditLog
  → 返回 {"code": "SUCCESS"} 给微信

POST /webhook/alipay
  → 验证签名（支付宝 RSA2 验签）
  → 提取 out_trade_no + trade_no + 金额
  → 同上事务逻辑
  → 返回 "success" 给支付宝
```

#### Product.stockQty 与 WarehouseStock 的关系

- `Product.stockQty` = 该商品所有仓库 `WarehouseStock.quantity` 的汇总（冗余字段，用于商品列表快速展示和库存预警判断）
- 真实库存以 `WarehouseStock` 为准，`Product.stockQty` 通过触发器或业务逻辑同步更新
- 扣减和锁定操作只操作 `WarehouseStock`，操作完成后异步更新 `Product.stockQty`

#### 并发扣减（乐观锁）

下单时的库存锁定采用乐观锁，防止超卖：

```
UPDATE warehouse_stock
SET locked_qty = locked_qty + :orderQty,
    quantity = quantity - :orderQty,
    version = version + 1
WHERE id = :stockId
  AND tenant_id = :tenantId
  AND version = :currentVersion
  AND quantity >= :orderQty
```

- version 不匹配或 quantity 不足时更新行数为 0，业务层捕获后返回"库存不足"
- 不使用悲观锁（SELECT FOR UPDATE），避免高并发时的锁等待

#### 超卖保护

- 第一道防线：下单前 Redis 预扣（DECRBY），快速拦截明显不足的请求
- 第二道防线：数据库乐观锁（如上），确保最终一致性
- 第三道防线：quantity 字段添加 CHECK 约束 `quantity >= 0`，数据库层兜底
- 支付超时释放：关单时反向操作 `quantity + lockedQty`，恢复库存

#### 多仓库发货策略

发货时按以下优先级选择仓库：

1. 就近发货：根据收货地址匹配最近仓库（Phase 2 实现，需要仓库地理坐标）
2. 优先级发货（Phase 1 默认策略）：按 `Warehouse.priority` 从小到大，选第一个库存充足的仓库
3. 拆单发货：单个仓库库存不足时，拆分到多个仓库发货（生成多条 Shipment）
4. 缺货处理：所有仓库库存总和不足时，标记订单为"待补货"，通知采购

### 6.8 MallCustomer 与 User 关系

#### 核心原则：员工和买家是两个完全独立的身份体系

```
User（员工）                     MallCustomer（买家）
├── 企业内部身份                  ├── 商城消费者身份
├── 手机号 + 企微 + 微信openid    ├── 微信openid + 手机号（可选）
├── RBAC 权限体系                 ├── 会员等级 + 积分 + 分销体系
├── 通过 Role 控制数据访问         ├── 通过 tenantId 隔离购物数据
└── 一个人只在一个租户下有一个User └── 同一openid在不同租户下有独立MallCustomer
```

#### 同一 openid 跨租户的处理规则

- 每个租户下独立创建 MallCustomer：同一个微信用户（openid 相同）在租户 A 和租户 B 分别有独立的 MallCustomer 记录
- 购物车隔离：Cart 关联 `tenantId + mallCustomerId`，不同租户的购物车完全独立
- 积分隔离：PointRecord 关联 `tenantId + mallCustomerId`，积分不跨租户流通
- 收货地址隔离：Address 关联 `tenantId + mallCustomerId`，每个租户独立维护
- 会员等级隔离：memberLevel 按租户独立计算，消费记录不跨租户累计
- 分销关系隔离：referrerId 只在同一租户内有效

### 6.9 Order vs MallOrder 生命周期

#### 两个订单模型的定位

- `Order`：ERP 同步进来的外部平台订单（抖店/天猫/美团等），通过 ERP 适配器写入，是外部数据的镜像
- `MallOrder`：自建商城内部产生的订单，由买家下单直接创建，是自有业务的原生订单

#### 双向同步机制

```
MallOrder → Order 方向（商城订单同步到工作站统一视图）：
  MallOrder 创建时 → 同步创建对应 Order 记录
  MallOrder 状态变更时 → 同步更新 Order.status
  关联外键：Order.mallOrderId → MallOrder.id
  Order.platformOrderId = "MALL-" + MallOrder.orderNo（加前缀区分来源）
  Order.shopId = 自建商城对应的 Shop 记录

Order → MallOrder 方向（不存在）：
  外部平台订单（ERP同步的 Order）没有对应 MallOrder
  外部订单只在 Order 表中存在
```

- 创建同步：MallOrder 在 Service 层创建时，同一事务中创建对应 Order
- 状态同步：MallOrder 状态变更时，通过 BullMQ 事件异步更新 Order.status（最终一致性）
- 冲突处理：以 MallOrder 为准（自有数据源），Order 是衍生视图

### 6.10 完整订单状态机

#### 枚举扩展（替代原有 OrderStatus）

```prisma
enum MallOrderStatus {
  pending_payment    // 待支付：订单已创建，等待买家支付
  paying             // 支付处理中：已调起支付渠道，等待回调
  paid               // 已支付：支付成功，等待商家发货
  pending_shipment   // 待发货：已支付，商家正在拣货打包
  shipped            // 已发货：已交物流，买家等待收货
  received           // 已签收：物流签收（自动或买家确认）
  completed          // 已完成：订单完结，超过售后期
  cancelled          // 已取消：买家取消或超时未支付自动取消
  after_sale         // 售后中：买家发起退货/退款申请
  refunding          // 退款处理中：退款已提交到支付渠道
  refunded           // 已退款：退款完成
  closed             // 已关闭：异常关闭（如风控拦截）
}
```

#### 状态流转全链路

正常流程：
- `pending_payment → paying`：买家点击支付，调起支付渠道
- `paying → paid`：支付回调确认成功
- `paying → pending_payment`：支付未完成（如用户取消支付弹窗），可重新支付
- `paid → pending_shipment`：系统自动流转，或商家确认备货
- `pending_shipment → shipped`：商家填写物流单号，触发发货
- `shipped → received`：物流签收回调，或买家手动确认收货
- `received → completed`：签收后 7 天（可配置）无售后申请，自动完结

取消流程：
- `pending_payment → cancelled`：买家主动取消，或超过 30 分钟未支付自动取消
- `paying → cancelled`：支付超时，BullMQ 延迟任务触发取消

售后流程：
- `received → after_sale`：买家发起退货/退款申请
- `shipped → after_sale`：物流途中买家申请退款（仅退款不退货场景）
- `after_sale → refunding`：售后审核通过，发起退款到支付渠道
- `refunding → refunded`：退款渠道回调确认
- `after_sale → received`：售后申请被拒绝或买家撤回，恢复原状态（保存在 MallAfterSale.previousStatus）

#### Order 表 OrderStatus 映射

MallOrder 的精细状态与 Order 表的粗粒度 OrderStatus 映射关系：

| MallOrderStatus | OrderStatus |
|-----------------|-------------|
| pending_payment, paying | _(不同步，Order 在支付成功后才创建)_ |
| paid, pending_shipment | paid |
| shipped, received | shipped |
| completed | completed |
| refunding, refunded | refunded |
| cancelled, closed | cancelled |

### 6.11 售后单完善

#### MallAfterSale 模型定义

```prisma
// 商城售后单：关联 MallOrder，支持退款/退货/换货
model MallAfterSale {
  id                  String       @id @default(uuid())
  tenantId            String
  afterSaleNo         String       // 售后单号（格式：AS-{yyyyMMddHHmmss}-{random4}）
  mallOrderId         String       // 关联 MallOrder.id
  mallCustomerId      String       // 关联 MallCustomer.id（发起售后的买家）
  orderItemId         String?      // 关联 OrderItem.id（具体哪个商品，整单退时为空）
  type                String       // refund_only / return_refund / exchange
  refundAmount        Decimal      @db.Decimal(12,2) // 申请退款金额
  reason              RefundReason // 复用已有枚举
  description         String?      // 买家补充说明
  evidence            Json?        // 图片URL、截图等举证材料
  previousStatus      String       // 发起售后前的订单状态（审核拒绝时用于恢复）

  // 物流信息（退货场景）
  returnTrackingNo    String?      // 退货物流单号
  returnCarrier       String?      // 退货物流公司
  returnAddress       Json?        // 退货收货地址

  // 退款关联
  refundPaymentId     String?      // 关联 RefundPayment.id（退款执行后写入）

  // 审核信息
  liability           Liability?   // 责任归属判定
  aiDecision          Json?        // ACI 客服中枢的 AI 判定结果
  reviewedBy          String?      // 审核人（User.id，人工审核时）
  reviewedAt          DateTime?
  reviewNote          String?      // 审核备注

  status              String       @default("pending")
  // 状态：pending（待审核）→ approved（审核通过）→ returning（退货中，等待买家寄回）
  //       → received_return（已收到退货）→ refunding（退款处理中）→ completed（完成）
  //       或 pending → rejected（审核拒绝）
  //       或 pending → cancelled（买家撤回）

  createdAt           DateTime     @default(now())
  updatedAt           DateTime     @updatedAt
  deletedAt           DateTime?

  @@unique([tenantId, afterSaleNo])
  @@index([tenantId, mallOrderId])
  @@index([tenantId, mallCustomerId])
  @@index([tenantId, status])
  @@index([tenantId, createdAt])
}
```

#### 售后状态流转

仅退款流程（refund_only）：
- `pending → approved → refunding → completed`：审核通过后直接发起退款
- `pending → rejected`：审核拒绝

退货退款流程（return_refund）：
- `pending → approved → returning → received_return → refunding → completed`
- `approved → returning`：审核通过，通知买家寄回商品，下发退货地址
- `returning → received_return`：商家确认收到退货
- `received_return → refunding`：触发退款到支付渠道（创建 RefundPayment）
- `refunding → completed`：退款成功回调

换货流程（exchange）：
- `pending → approved → returning → received_return → reshipping → completed`
- `received_return → reshipping`：收到退货后重新发货
- `reshipping → completed`：买家确认收到换货

#### 与 AfterSale（ERP 售后单）的关系

- `MallAfterSale` 是自建商城的售后单，由买家在小程序发起
- `AfterSale` 是 ERP 同步的外部平台售后单（抖店/天猫等）
- 两者独立存在，不互相关联
- MallAfterSale 审核完成后，可通过 BullMQ 异步同步到 AfterSale（用于统一售后统计报表）

### 6.12 异步任务框架

#### 技术选型：BullMQ + Redis

使用 BullMQ（基于 Redis 的 Node.js 任务队列）作为异步任务框架，不引入 RabbitMQ/Kafka 等额外中间件，符合最小架构原则。

#### Queue 定义

```typescript
// jobs/queues.ts — 所有队列定义

// 支付相关队列
const paymentQueue = new Queue('payment', { connection: redis })
// Job 类型：
//   payment:close    — 超时未支付关单（延迟任务，30分钟后触发）
//   payment:query    — 支付结果主动查询（延迟任务，5分钟后触发）
//   payment:refund   — 发起退款请求

// 订单相关队列
const orderQueue = new Queue('order', { connection: redis })
// Job 类型：
//   order:auto-complete   — 签收后自动完结（延迟任务，7天后触发）
//   order:sync-to-erp     — MallOrder 状态同步到 Order 表
//   order:auto-cancel     — 超时未支付取消（与 payment:close 配合）

// 库存相关队列
const stockQueue = new Queue('stock', { connection: redis })
// Job 类型：
//   stock:release       — 取消/关单时释放锁定库存
//   stock:sync-total    — 重算 Product.stockQty（WarehouseStock 变更后）
//   stock:warning       — 库存预警检查（定时任务，每小时）

// 数据同步队列
const syncQueue = new Queue('sync', { connection: redis })
// Job 类型：
//   sync:erp-incremental — ERP 增量同步（每5分钟）
//   sync:erp-full        — ERP 全量同步（每天凌晨2点）
//   sync:erp-manual      — 手动触发同步

// 通知队列
const notifyQueue = new Queue('notify', { connection: redis })
// Job 类型：
//   notify:wechat-template — 微信模板消息（发货/退款/签收通知）
//   notify:admin-alert     — 管理员告警（异常/库存不足/大额退款）

// AI 任务队列
const aiQueue = new Queue('ai', { connection: redis })
// Job 类型：
//   ai:daily-report      — 每日早报生成
//   ai:anomaly-check     — 异常检测巡检
//   ai:knowledge-index   — 知识库向量化索引
```

#### 死信队列（Dead Letter Queue）

所有队列统一配置死信处理：

```typescript
// 全局默认配置
const defaultJobOptions = {
  attempts: 3,                          // 最多重试 3 次
  backoff: { type: 'exponential', delay: 5000 }, // 指数退避：5s → 10s → 20s
  removeOnComplete: { count: 1000 },    // 保留最近 1000 条成功记录
  removeOnFail: false,                  // 失败记录不自动删除
}

// 失败超过 maxAttempts 的 Job 进入死信队列
const deadLetterQueue = new Queue('dead-letter', { connection: redis })

// 全局 failed 事件监听 → 写入死信 + 告警
worker.on('failed', (job, err) => {
  if (job.attemptsMade >= job.opts.attempts) {
    deadLetterQueue.add('dead', {
      originalQueue: job.queueName,
      jobId: job.id,
      data: job.data,
      error: err.message,
      failedAt: new Date(),
    })
    // 告警管理员
    notifyQueue.add('notify:admin-alert', {
      type: 'dead_letter',
      detail: `Job ${job.name} failed after ${job.attemptsMade} attempts`,
    })
  }
})
```

#### 幂等性保证

所有 Job 处理函数必须满足幂等性要求：

- 支付关单：先检查 Payment.status，已为 closed/paid 则跳过
- 库存释放：使用 `WHERE locked_qty >= :releaseQty` 防止重复释放
- 订单同步：通过 `updatedAt` 时间戳比对，旧数据不覆盖新数据
- 通知发送：通过 `Notification` 表的唯一键（tenantId + userId + type + resourceId）去重

#### 失败重试策略

| 队列 | 重试次数 | 退避策略 | 超时 | 说明 |
|------|---------|---------|------|------|
| payment | 5 | 指数退避 5s→160s | 30s | 支付关键路径，多重试 |
| order | 3 | 指数退避 5s→20s | 60s | 订单状态最终一致 |
| stock | 3 | 指数退避 3s→12s | 15s | 库存操作要快 |
| sync | 3 | 指数退避 60s→240s | 300s | ERP 同步可容忍延迟 |
| notify | 2 | 固定间隔 30s | 15s | 通知非关键路径 |
| ai | 2 | 固定间隔 60s | 120s | AI 调用容忍延迟 |

### 6.13 密钥轮换策略

#### AES-256-GCM 密钥版本化

加密字段存储格式：`{keyVersion}:{iv}:{ciphertext}:{authTag}`

- `keyVersion`：当前使用的密钥版本号（如 v1、v2）
- 密钥版本映射存储在阿里云 KMS，每个版本对应一个独立的数据密钥
- 新数据加密时始终使用最新版本密钥
- 解密时根据密文前缀中的 keyVersion 选择对应密钥

#### 密钥轮换流程

```
1. 在 KMS 中创建新版本数据密钥（如 v2）
2. 更新应用配置：ENCRYPTION_KEY_VERSION=v2
3. 新数据自动用 v2 加密
4. 旧数据（v1 加密）仍可正常解密（读取时识别 keyVersion）
5. 后台任务（BullMQ）逐批重加密旧数据：
   - 读取 v1 加密的记录 → 用 v1 密钥解密 → 用 v2 密钥重加密 → 更新记录
   - 批量处理，每批 100 条，间隔 1 秒，避免数据库压力
   - 记录迁移进度到 Redis，支持断点续传
6. 全部迁移完成后，废弃 v1 密钥（KMS 中标记 disabled）
```

#### 轮换周期

- 常规轮换：每 90 天执行一次密钥轮换
- 紧急轮换：密钥疑似泄露时立即执行
- 重加密窗口：轮换后 7 天内完成所有旧数据重加密

#### KMS 不可用降级策略

- 正常运行：所有加解密通过 KMS 获取数据密钥（信封加密）
- KMS 短暂不可用（< 5分钟）：使用本地缓存的数据密钥继续工作（缓存 TTL = 5分钟）
- KMS 持续不可用（> 5分钟）：
  - 写操作（新数据加密）：返回 503 + 告警管理员，不使用明文存储
  - 读操作（解密已有数据）：使用本地缓存密钥继续解密，直到缓存过期
  - 缓存过期后：返回加密原文，标记为 `[ENCRYPTED]`，前端展示"数据暂时不可用"
- KMS 恢复后：自动刷新本地缓存，恢复正常加解密

#### 涉及加密的字段清单

| 模型 | 字段 | 内容 |
|------|------|------|
| ERPConnection | credentials | ERP 平台 API Key/Secret |
| Supplier | bankInfo | 供应商银行卡号、开户行 |
| CustomerServiceChannel | config | 客服渠道 API Key/Webhook Secret |

### 6.14 数据模型设计规范

- **多租户隔离**：所有表必须包含 `tenantId`，所有查询必须带 `tenantId` 过滤
- **软删除**：核心业务表均包含 `deletedAt DateTime?`，禁止物理删除
- **时间戳**：核心表均包含 `createdAt` + `updatedAt`，审计和消息等追加型表仅 `createdAt`
- **时区规范**：所有时间字段统一存储 **UTC 时间**，展示时按租户设置的时区（默认 Asia/Shanghai）转换。ERP 数据同步时将各平台本地时间转换为 UTC 后入库。前端请求日期范围参数统一使用 ISO8601 格式（如 `2026-03-29T00:00:00+08:00`），后端统一转 UTC 处理
- **金额精度**：所有金额字段统一 `Decimal(12,2)`，不使用 Float
- **枚举类型**：业务状态优先使用 Prisma enum，减少魔法字符串

## 十二、开发规范

1. **所有 API 响应统一格式**：`{ success, data, error?, meta? }`
2. **所有查询必须带 tenant_id**：在 middleware 层自动注入
3. **敏感字段加密**：bank_info、credentials 使用 AES-256-GCM（详见第十三节安全设计）
4. **操作审计**：财务操作、结算确认、代码部署必须写 AuditLog
5. **错误处理**：所有错误返回标准 error code，不暴露内部细节
6. **测试覆盖**：核心业务逻辑覆盖率 ≥ 80%
7. **软删除**：核心业务数据不可直接删除，统一使用 `deletedAt` 软删除字段
8. **输入校验**：所有 API 输入必须通过 Zod schema 校验
9. **时区统一**：所有时间存储 UTC，展示按租户时区转换，ERP 数据入库时转 UTC

## 十三、安全设计

### 13.1 认证方案（JWT）

| 配置项 | 值 | 说明 |
|-------|------|------|
| 签名算法 | RS256 | 非对称签名，公钥可安全分发给各服务验证 |
| Access Token 有效期 | 15 分钟 | 短期令牌，减少泄露风险 |
| Refresh Token 有效期 | 7 天 | 长期令牌，存储在 httpOnly cookie |
| Token 黑名单 | Redis SET | 用户登出、角色变更时将 token jti 加入黑名单 |
| 角色变更即时失效 | 是 | 管理员修改用户角色后，该用户所有已发 access token 立即失效 |

Token 刷新流程：
1. 客户端携带 access token 请求 API
2. access token 过期时返回 401
3. 客户端使用 refresh token 请求 `/auth/refresh`
4. 服务端用 SHA-256(refresh_token) 查询 `RefreshToken` 表，验证未过期、未吊销
5. 签发新 access token + 新 refresh token，旧 refresh token 标记 `revokedAt`（Rotation）
6. 若检测到已吊销的 refresh token 被复用（重放攻击），立即吊销该用户所有 refresh token

### 13.2 微信小程序登录流程

```
用户点击登录
  → wx.login() 获取 code
  → 前端将 code 发送到后端 /auth/wechat-login
  → 后端调用微信 code2session 接口换取 session_key + openid
  → 查询 User 表是否存在该 openid
    → 存在：直接签发 JWT（access token + refresh token）
    → 不存在：引导用户绑定手机号（wx.getPhoneNumber）
      → 后端解密手机号 → 创建 User 记录 → 签发 JWT
  → 返回 JWT 给前端，前端存入本地存储
```

注意事项：
- `session_key` 不可下发给前端，仅服务端保留用于解密
- 手机号解密使用微信提供的 AES-128-CBC 方案
- 首次登录后 `wechatOpenid` 字段写入 User 表，后续登录直接匹配

### 13.3 加密方案

| 项目 | 方案 | 说明 |
|------|------|------|
| 加密算法 | AES-256-GCM | 认证加密，同时保证机密性和完整性 |
| 密钥管理 | 阿里云 KMS | 主密钥托管在 KMS，应用层获取数据密钥（信封加密） |
| IV 生成 | 每次加密随机生成 12 字节 IV | 与密文一起存储，格式：`iv:ciphertext:authTag` |
| 加密字段 | ERPConnection.credentials、Supplier.bankInfo | 所有含密码、密钥、银行卡号的字段 |

### 13.5 输入校验

所有 API 输入必须通过 Zod schema 校验，在 controller 层入口统一执行：

```typescript
// 示例：创建订单的输入校验
const CreateOrderSchema = z.object({
  shopId: z.string().uuid(),
  platformOrderId: z.string().min(1).max(64),
  totalAmount: z.number().positive().multipleOf(0.01),
  items: z.array(z.object({
    skuCode: z.string(),
    qty: z.number().int().positive(),
    price: z.number().positive(),
  })).min(1),
});
```

规则：
- 字符串字段限制最大长度，防止存储攻击
- 数值字段限制范围，金额必须为正数
- 数组字段限制最大元素数量
- 校验失败返回 400 + 详细字段错误信息

### 13.6 Rate Limiting

| 限流维度 | 限制 | 说明 |
|---------|------|------|
| 租户级 | 100 req/min | 防止单个租户影响平台整体 |
| 用户级 | 30 req/min | 防止单用户滥用 |
| AI 对话接口 | 10 req/min（用户级） | AI 调用成本高，单独限流 |
| 登录接口 | 5 req/min（IP级） | 防止暴力破解 |

实现：基于 Redis 滑动窗口算法，超限返回 `429 Too Many Requests`。

## 十四、数据备份与灾备

### 14.1 备份目标

| 指标 | 目标值 | 说明 |
|------|-------|------|
| RPO（恢复点目标） | < 1 小时 | 最多丢失 1 小时数据 |
| RTO（恢复时间目标） | < 4 小时 | 4 小时内恢复服务 |

### 14.2 备份策略

| 备份类型 | 频率 | 保留时间 | 存储位置 |
|---------|------|---------|---------|
| WAL 归档 | 每小时 | 7 天 | 阿里云 OSS |
| 全量备份（pg_dump） | 每天凌晨 3 点 | 30 天 | 阿里云 OSS |
| 月度归档 | 每月 1 号 | 1 年 | 阿里云 OSS（低频存储） |

### 14.3 备份执行

```bash
# WAL 归档配置（postgresql.conf）
archive_mode = on
archive_command = 'ossutil cp %p oss://backup-bucket/wal/%f'

# 每日全量备份（crontab）
0 3 * * * pg_dump -Fc enterprise_workstation | ossutil cp - oss://backup-bucket/daily/$(date +\%Y\%m\%d).dump
```

### 14.4 恢复演练

- **频率**：每月一次
- **流程**：从 OSS 拉取最近备份 → 恢复到独立测试实例 → 验证数据完整性和业务查询 → 记录恢复耗时
- **演练结果**：记录到运维文档，恢复耗时超过 RTO 需启动优化
