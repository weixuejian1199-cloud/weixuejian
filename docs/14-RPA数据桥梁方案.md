# 14 - RPA 数据桥梁方案

> v1.0 · 2026-04-02
> 决策人：创始人
> 制定人：CTO

---

## 一、决策背景

### 问题
1. 聚水潭API只能获取订单数据，**拿不到财务数据**（平台结算单、应收款、佣金明细、推广费用等）
2. 各平台API申请进度不一，小红书/美团API对小商家几乎不开放
3. 财务板块（会计/出纳工作台）需要完整的多平台经营数据才能做账
4. 运营板块需要各平台精细化数据（生意参谋/千川/抖店罗盘等）用于决策

### 决策（ADR-031）
采用 **RPA 数据桥梁方案**，通过浏览器自动化采集各平台数据，作为API的前置替代方案。

**核心原则：只读不写。** RPA只用于数据采集，一切写操作（改价、发货、回评、投放调整）一律不碰。

### 2.3 权限隔离

```
管理员(admin/super_admin)：
  ├── 可以：绑定/解绑平台账号、查看RPA运行状态、手动触发数据同步
  └── 看到：「系统设置 → 平台授权管理」

财务(finance)：
  ├── 可以：查看所有平台的财务数据（结算单、应收款、佣金）
  ├── 不能：看到平台账号信息、操作RPA设置
  └── 体验：打开会计工作台，各平台数据已自动就位，直接做账

运营(operation)：
  ├── 可以：查看各平台经营数据（GMV、UV、转化率、投放ROI）
  ├── 不能：看到平台账号信息、操作RPA设置
  └── 体验：打开运营工作台，概览页数据自动聚合，无感知RPA存在

原则：RPA对业务岗位完全透明，他们不知道也不需要知道数据是API来的还是RPA来的。
```

## 三、RPA采集范围（只读清单）

### 3.1 财务数据（会计/出纳工作台用）

| 数据 | 来源平台 | 采集频率 | 用途 |
|------|---------|---------|------|
| 平台结算单 | 抖店/天猫/美团 | 每日 | 平台对账、生成凭证 |
| 佣金/技术服务费明细 | 抖店/天猫/美团 | 每月 | 成本核算、费用归集 |
| 推广费用明细 | 千川/直通车/引力魔方 | 每日 | 推广费归集、ROI核算 |
| 退款/扣款明细 | 抖店/天猫 | 每日 | 退货退款凭证 |
| 保证金/押金变动 | 各平台 | 每周 | 其他应收款 |
| 发票开具记录 | 各平台 | 每月 | 销项发票核对 |

### 3.2 运营数据（运营工作台用）

| 数据 | 来源平台 | 采集频率 | 用途 |
|------|---------|---------|------|
| 每日经营概况（GMV/UV/转化率） | 抖店罗盘/生意参谋 | 每日 | 概览KPI |
| 商品销量/库存 | 各平台后台 | 每小时 | 库存预警、热销排行 |
| 投放计划效果数据 | 千川/万相台 | 每日 | 投放管理看板 |
| 评价/差评内容 | 各平台 | 每日 | 评价管理、差评预警 |
| 竞品商品页（价格/销量/评分） | 各平台前台 | 每日 | 竞品监控 |
| 搜索关键词排名 | 生意参谋 | 每日 | SEO优化 |

### 3.3 绝对不碰的操作（红线）

| 操作 | 原因 |
|------|------|
| 修改商品价格 | 改错了影响销售和品牌 |
| 修改库存数量 | 超卖风险 |
| 发货/填写物流单号 | 发错货不可逆 |
| 回复评价/客服消息 | 品牌形象风险 |
| 创建/修改投放计划 | 涉及资金 |
| 报名平台活动 | 需要人工判断 |
| 修改店铺设置 | 安全风险 |

## 四、技术选型

### 4.1 采集引擎

| 方案 | 优点 | 缺点 | 推荐 |
|------|------|------|------|
| **Playwright (Node.js)** | 与项目技术栈一致、反检测能力强、跨浏览器 | — | **推荐** |
| Puppeteer | 成熟稳定、社区大 | 仅支持Chrome | 备选 |
| Selenium | 最老牌 | 慢、容易被检测 | 不推荐 |

选择 **Playwright**：
- 与我们 Node.js + TypeScript 技术栈完全匹配
- 内置反检测能力（stealth mode）
- 支持 Chromium/Firefox/WebKit
- 可以跑在 ECS 服务端（Headless模式）

### 4.2 部署方式

```
Phase 1（起步）：
  管理员电脑安装 Electron 桌面客户端
  内嵌 Playwright + 采集脚本
  数据通过 HTTPS 同步到 ECS
  
  优点：cookie在本地，安全性高
  缺点：需要电脑开着

Phase 2（稳定后）：
  迁移到 ECS 服务端运行（Headless）
  管理员定期刷新 cookie（或用长效token）
  24小时自动采集
  
  优点：全自动，无人值守
  缺点：cookie过期需处理
```

### 4.3 数据存储

```
RPA采集的原始数据
      │
      ▼
  PostgreSQL（与工作站同库）
      │
      ├── rpa_platform_accounts     -- 平台授权账号（加密存储）
      ├── rpa_sync_logs             -- 采集日志（时间/状态/错误）
      ├── rpa_raw_data              -- 原始采集数据（JSON）
      │
      ▼
  数据标准化后写入业务表
      │
      ├── platform_settlements      -- 平台结算单
      ├── platform_commissions      -- 佣金明细
      ├── daily_channel_metrics     -- 每日渠道数据
      ├── ad_campaign_data          -- 投放数据
      └── competitor_snapshots      -- 竞品快照
```

```

                 │                          │                    │

天猫订单     RPA采集 ──────────→ 申请到API ──→ 切换为API调用
抖店结算单   RPA采集 ──────────→ RPA采集 ────→ 看API是否开放
天猫结算单   RPA采集 ──────────→ RPA采集 ────→ 看API是否开放
千川数据     RPA采集 ──────────→ 申请到API ──→ 切换为API调用
小红书数据   RPA采集 ──────────→ RPA采集 ────→ RPA采集（API不太可能开放）

竞品数据     RPA采集 ──────────→ RPA采集 ────→ RPA采集（永远RPA）

切换原则：
  DataSourceAdapter 统一接口

  └── 切换时只改适配器配置，业务层无感知
```

## 六、风险控制

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 平台检测到RPA封号 | 低（只读+低频） | 高 | 控制频率、模拟人类行为、不做写操作 |
| Cookie过期 | 必然（定期发生） | 低 | 监控cookie有效性，过期时通知管理员重新登录 |
| 平台页面改版 | 中（每季度可能） | 中 | 采集脚本模块化，改版时只改对应平台脚本 |
| 数据采集遗漏 | 低 | 低 | 采集完成后校验数据完整性，异常告警 |

### 降低封号风险的措施

1. **绝对只读** — 不做任何写操作
2. **合理频率** — 每日1-2次全量采集，不高频轮询
3. **人类行为模拟** — 随机间隔、鼠标轨迹、滚动
4. **非高峰时段** — 凌晨2-5点执行采集
5. **一个账号一个IP** — 不复用IP跨账号

## 八、与现有架构的关系

```
现有：MallAdapter → ztdy-open API → 商城数据
新增：RPAAdapter  → Playwright   → 平台经营/财务数据

两者并列，都通过 DataSourceAdapter 统一接口对外提供数据。
业务层（财务/运营/AI Agent）不关心数据来源是API还是RPA。

未来：某个平台API开放后
  RPAAdapter.douyin → 替换为 → APIAdapter.douyin
  接口不变，业务层零修改
```

这就是 ADR-024（Phase 1 数据策略）的延伸：
- **主数据源**：MallAdapter（ztdy-open）— 已有
- **补充数据源**：RPAAdapter（Playwright）— 本方案新增
- **辅助数据源**：Excel/CSV上传 — 已有
