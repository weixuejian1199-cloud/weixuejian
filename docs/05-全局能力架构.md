## 一、产品定位

### 三句话说清楚
1. **对员工**：打开小程序，问一句话，就能完成工作（查数据、出报表、对账、报税）
2. **对老板**：一个驾驶舱看全局，一句话管公司
3. **对外**：SaaS 产品，卖给其他电商企业

### 核心价值（v2.0 明确）
核心价值不是替代金蝶或七鱼的单点功能深度，而是**用AI把多个系统串起来**——老板一句话完成原来需要打开5个系统才能干的事。Phase 1 追求跨模块AI串联体验，不追求单模块功能深度。

### Phase 1 数据策略（ADR-024）
- **主数据源**：MallAdapter（ztdy-open API）— 现有商城146万用户/95万订单的真实数据
- **辅数据源**：Excel/CSV上传 — 覆盖无API权限的租户/岗位
- Phase 1 商品/订单从API实时读取不落库，Phase 2自建商城后切换为同库直查

## 五、工具市场架构（核心扩展机制）

### 5.1 核心设计理念

**一个工具 = 一个独立 Agent = 独立记忆 = 独立知识库 = 独立能力**

**员工所有操作，都通过对话完成。不做表单，不做配置页。**

```
传统做法（我们不做）：
  工具 → 复杂表单 → 员工手动填写 → 容易出错 → 需要培训

我们的做法：
  工具 → 就是一个AI对话 → 员工用自然语言说需求 → AI理解+确认+执行

前端只有一个核心组件：聊天界面
  主页 Tab3 "AI工作站"  → 连接 Master Agent（路由到子Agent）
  工具页 每个工具点进去  → 连接 该工具的独立 Agent

  同一个聊天组件，不同的Agent，不同的记忆，不同的知识库。
```

**举例说明这个设计的威力：**

| 场景 | 员工操作 | 背后发生的事 |
|------|---------|-------------|
| 配置小酮回复风格 | 跟小酮说"回复控制在50字以内" | Agent 理解→确认→更新 config |
| 添加知识库 | 跟小酮说"新产品抹茶味奶茶，MCT-C8+抹茶粉" | Agent 整理→确认→写入知识库→重新索引 |
| 运营分析 | 把数据文件发给运营助手，说"帮我分析上周趋势" | Agent 解析文件→分析→生成报告 |
| 对账 | 把供应商结算单发给对账工具，说"帮我核对3月的账" | Agent 解析→匹配→计算差异→生成结算单 |
| 清洗数据 | 把脏数据文件发给清洗工具，说"去重+修复格式" | Agent 识别问题→清洗→确认→输出干净数据 |

```
初期（5-10个工具）        中期（20-50个）          长期（100+个）
├── AIBI小酮(营养师)      ├── 按分类组织            ├── 工具搜索
├── 脏数据清洗            │   ├── 健康咨询类        ├── 标签筛选
├── 电商运营助手          │   ├── 数据类            ├── 第三方工具接入
├── 供应商对账            │   ├── 营销类            ├── SaaS租户自建工具
└── 报税助手              │   ├── 财务类
                          │   └── 系统类
注：客服不在工具市场中，客服是独立子系统（见4.3节）
```

### 5.4 架构师建议：在你的基础上的增强方案

你的核心思路已经非常好，我从架构角度补充三个增强点：

**增强1：工具 Agent 的三种对话模式**

每个工具 Agent 不是只有一种对话模式，而是根据用户意图自动切换：

```
┌─────────────────────────────────────────────────────────┐
│  模式1：使用模式（默认）                                 │
│  员工把数据/文件/问题丢给工具，工具执行核心能力          │
│  "帮我分析上周销售数据" → 分析 + 输出报告                │
│  "这个客户问生酮能不能吃火锅" → 查知识库 + 生成回复      │
│                                                         │
│  模式2：配置模式（员工说"调整"/"修改"/"设置"时触发）     │
│  员工修改工具行为，Agent 理解→确认→执行                  │
│  "回复控制在50字" → 理解 → 展示变更 → 确认 → 生效        │
│  "加一条知识：新品抹茶味" → 整理 → 预览 → 确认 → 写入    │
│                                                         │
│  模式3：测试模式（员工说"测试"/"试一下"/"模拟"时触发）   │
│  员工验证修改效果，不影响线上                             │
│  "模拟一下用户问头疼怎么办" → 用新配置生成回复 → 展示     │
│  "对比一下修改前后的回复" → 新旧对比展示                  │
└─────────────────────────────────────────────────────────┘

三种模式同一个对话界面，Agent 自动识别切换，用户无感。
```

**增强2：工具之间可以协作**

工具不是完全孤立的。Master Agent 可以串联多个工具：

```
主页 AI 对话中：
  员工："帮我出一份本周运营周报，包含销售数据和供应商结算情况"

  Master Agent 自动编排：
  → 调用"运营助手"获取销售数据和趋势
  → 调用"供应商对账"获取结算汇总
  → 汇总两个工具的结果，生成周报

工具独立时是独立 Agent，但通过 Master Agent 可以组合使用。
```

**增强3：工具模板 → 快速创建新工具**

当工具数量增多后，很多工具的基础结构是相似的（客服类、分析类、生成类）。
建议设计工具模板机制：

```
老板在驾驶舱说："帮我创建一个护肤顾问工具，参考小酮的架构"

System Agent 自动：
1. 复制 AIBI小酮的 Agent 基础结构
2. 清空知识库（等待填充护肤知识）
3. 调整人格设定（护肤顾问 vs 营养师）
4. 创建 ToolDefinition + ToolInstance
5. 新工具立刻出现在工具列表中

模板类型：
```

## 九、财务模块建设策略

### 核心决策：自建为主，不二开，不用金蝶/用友

```
为什么自建而非二开 ERPNext？
├── ERPNext 是 Python + MariaDB，我们是 TypeScript + PostgreSQL，技术栈不同
├── 它的财务模块和 ERP 深度耦合，拆不干净
├── 中国会计准则适配做得粗糙，还是要自己写
├── 二开等于维护两套技术栈，后续升级是噩梦
└── 电商财务场景比制造业简单得多，自建完全可行

为什么不用金蝶/用友？
├── 公司目前没有在用，没有历史包袱
├── 用了就多一个依赖和年费
└── 我们的 AI 自动做凭证 比手工录入金蝶更先进
```

| 来源 | 怎么用 | 获取方式 |

### 13.2 自建商城的核心优势（ADR-022）

```
自建商城 = API 你说了算 = AI 能控制一切

传统商家：
  淘宝/抖店后台 → 平台不开放 → AI 只能看不能动
  有赞/微盟商城 → API 有限 → 只能做一部分

你的做法：
  自建商城 + 共享数据库 → AI 工作站直接控制商城所有功能
  老板说"把新品上架定价89" → Operation Agent 直接执行
  系统检测到库存不足 → 自动下架 → 通知采购补货
  退货率超标 → 自动降权 → 通知运营排查
```

#### AI 擅长 vs 传统界面擅长

| AI 助手擅长（对话更快） | 传统界面擅长（表格/按钮更快） |
|----------------------|---------------------------|
| 深度分析："对比上周数据" | 浏览列表：快速扫一眼904个商品 |
| 批量操作："过期商品全部下架" | 单条操作：点某个商品的"编辑" |
| 跨数据关联："退货率高的供应商" | 图片确认：看商品主图排版是否OK |
| 自然语言查询："库存低于50的商品" | 拖拽排序：调整Banner顺序 |

**不互斥**——传统列表页能做的事（搜索、筛选、翻页、点按钮），AI 助手也能做但不一定更快。两个入口并存，用户自己选更顺手的。

### 13.4 商城数据模型融合策略（ADR-024/025 已确认）

> **权威数据源**：所有商城表的字段定义、状态机、关联关系以 `01-企业AI工作站-主文档.md` 第六节为准。本节只描述融合策略概要。

**三个核心拆分决策：**

| 决策 | 内容 | 原因 |
|------|------|------|
| 员工/买家拆分 | User（员工）与 MallCustomer（买家）独立，同一 openid 每租户一条 MallCustomer | 权限体系完全不同，购物车/积分/地址按租户隔离 |
| 订单拆分 | Order（ERP外部同步）与 MallOrder（商城内部产生）独立，通过 Order.mallOrderId 关联 | 来源和生命周期不同，MallOrder 有完整12状态流转 |
| 共享表合并 | Supplier/Settlement/Product 在原有基础上扩展字段，不新建表 | 同一数据库同一 Schema，避免冗余 |

**新增商城相关表（约 22 张，含支付/库存/售后）：**

- 买家体系：MallCustomer、Address、PointRecord、MemberLevelHistory
- 商城交易：MallOrder、OrderItem、Cart、MallAfterSale
- 支付体系：Payment、RefundPayment（详见01-主文档 6.6 支付流程设计）
- 商品扩展：SKU、Category、ProductReview、AIProductScore
- 库存体系：Warehouse、WarehouseStock（详见01-主文档 6.7 库存管理设计）
- 物流：Shipment
- 分销体系：RewardRecord、BonusPool、WithdrawalRecord
- 营销：Coupon、CouponRecord
- 配置：HomepageConfig

**共享原则：**
- 同一个 PostgreSQL + 同一个 Prisma Schema
- 所有商城表都有 tenantId（多租户从第一天设计进去）
- AIBIAINI 品牌 = 创始租户，未来 SaaS 客户直接能用商城功能
- 异步任务统一使用 BullMQ（详见01-主文档 6.12 异步任务框架）
- 加密字段密钥轮换策略见01-主文档 6.13
