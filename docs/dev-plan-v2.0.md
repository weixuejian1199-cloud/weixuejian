# ATLAS AI 能力升级 v2.0 — 开发思路方案

> 状态：**待审核**，审核通过后方可开始编码。

---

## 一、现状盘点

### 1.1 核心入口文件

| 文件 | 职责 | 行数 |
|------|------|------|
| `server/atlas.ts` | 文件上传解析、场景识别、System Prompt、AI 对话、suggestedActions 生成 | ~1400 |
| `server/hr.ts` | HR 模块独立路由：工资条上传/生成/下载、考勤上传/分析/下载 | ~650 |
| `client/src/pages/MainWorkspace.tsx` | 主对话区：文件拖入、进度展示、消息渲染、快捷按钮展示 | ~1500 |
| `client/src/pages/HRCenterPage.tsx` | HR 中心入口页：两张功能卡片（工资条/考勤），点击进入各自子流程 | ~116 |

### 1.2 已实现的能力（现状）

**场景识别（`detectScenario`，atlas.ts L266）**
- 已有：payroll / attendance / dividend / sales / inventory / general 六类场景
- 通过字段名正则打分，识别置信度 0-1
- **不足**：识别后只用于生成 suggestedActions，未对缺失字段做容错处理

**数据质量检测（atlas.ts L713）**
- 已有：缺失值检测（>5% 警告，>20% 重点提示），输出 `qualityIssues` 字符串
- **不足**：只做提示，不做容错计算（缺失字段不会按 0 处理继续汇总）

**字段同义词映射（atlas.ts L1132）**
- 已有：在对话 System Prompt 里有一段 `fieldAliasContext`，列出别名映射
- **不足**：仅靠 AI 理解，没有代码层面的标准化（跨文件合并时字段名不统一会导致合并失败）

**HR 工资条（hr.ts L110）**
- 已有：AI 识别字段（`detectPayslipFields`）+ 个税计算（`calcTax`）+ 生成三 Sheet Excel
- 已有：前端 HRCenterPage 有入口卡片，点击进入上传流程
- **不足**：
  - 没有"主动感知"：用户在主对话区上传工资表，不会自动弹出"生成工资条"快捷按钮（虽然 suggestedActions 里有"生成工资条"按钮，但点击后是发一条 AI 消息，不是直接跳转 HR 流程）
  - 没有从文件名/日期列自动提取工资期间
  - 字段同义词覆盖不够（如"固定工资"未映射到"底薪"）

**HR 考勤（hr.ts L160）**
- 已有：AI 识别字段（`detectAttendanceFields`）+ 迟到/早退/旷工计算 + 部门汇总
- **不足**：
  - 加班计算逻辑缺失（`hr.ts` 中无加班费率计算）
  - 异常标记只在 Excel 中，对话框内没有高亮提示
  - HRCenterPage 的考勤入口不够显眼（和工资条并列，没有独立快捷入口）

**财务场景**
- 已有：System Prompt 中有借贷/利润/费用等财务逻辑描述，靠 AI 理解
- **不足**：
  - 没有代码层面的借贷平衡校验（`debit === credit` 强制检查）
  - 所有数值计算用 JS 原生 `Number`，存在浮点误差（如 `0.1 + 0.2 = 0.30000000000000004`）

## 二、改动方案

### Phase 1（本周，优先级最高）

#### P1-A：出纳场景 — 智能字段对齐 + 缺失值容错

**改哪里**：`server/atlas.ts`，新增 `normalizeFieldNames()` 函数

**具体做法**：
1. 在 `detectScenario` 之前，新增一个**代码层面的字段标准化函数**，将常见同义词统一映射到标准字段名：
   ```
   "销售额" | "订单金额" | "收入" | "实收" | "流水" → "总销售额"
   "手续费" | "服务费" | "平台费" → "手续费"
   "退款" | "退货金额" | "退单" → "退款金额"
   ```
2. 标准化后，若某行/某文件缺少"手续费"或"退款"字段，**自动补 0**，不中断计算
3. 在 `qualityIssues` 中追加一条："检测到 X 个店铺缺少[手续费]字段，已按 0 处理"

**影响评估**：仅新增函数，不修改现有逻辑，风险极低

---

#### P1-B：HR 中心 — 文件自动识别 + 一键生成工资条

**改哪里**：`server/atlas.ts`（suggestedActions 生成逻辑）+ `client/src/pages/MainWorkspace.tsx`（快捷按钮点击处理）

**具体做法**：
1. 在 `atlas.ts` 的 suggestedActions 生成逻辑中，当 `scenario.type === "payroll"` 时，将"生成工资条"按钮的 `prompt` 改为一个特殊标识符 `__HR_PAYSLIP__`，而不是普通 AI 消息
2. 在 `MainWorkspace.tsx` 的快捷按钮点击处理中，检测到 `__HR_PAYSLIP__` 时，**直接跳转到 HR 中心工资条流程**（`setActiveNav("hr")`），并携带当前文件的 session_id
3. 同理，`scenario.type === "attendance"` 时，"考勤汇总"按钮触发 `__HR_ATTENDANCE__`，跳转到考勤流程

**智能预填工资期间**：
- 在 `atlas.ts` 文件解析时，从文件名（正则匹配 `2026年3月`、`202603`、`2026-03` 等格式）或数据中的日期列提取期间
- 提取结果通过 `suggested_actions` 的 `meta` 字段传到前端，HR 流程自动预填

**影响评估**：修改 suggestedActions 的 prompt 格式（新增特殊标识符），需同步更新前端的点击处理逻辑；不影响普通 AI 对话流程

### Phase 2（下周）

#### P2-A：财务场景 — 借贷平衡检查

**改哪里**：`server/atlas.ts`，在 `detectScenario` 中新增 `finance` 场景类型

**具体做法**：
1. 新增场景类型 `finance`：检测字段中是否含"借方"/"贷方"/"debit"/"credit"
2. 若识别为财务凭证，在 `qualityIssues` 中强制执行：`sum(借方) vs sum(贷方)`，差额 ≠ 0 时输出："借贷不平衡：借方总额 X，贷方总额 Y，差额 Z，请检查"
3. 同时检测是否缺少对应科目（如只有收入没有税金）

**影响评估**：新增场景类型，不影响现有 payroll/attendance/sales 场景

---

#### P2-B：财务场景 — 高精度计算

**改哪里**：`server/atlas.ts` 中的数值计算部分（`computeKeyMetrics`、`calcTax`）

**具体做法**：
1. 引入 `decimal.js`（或 `big.js`）替换涉及金额的 `Number` 运算
2. 重点覆盖：汇总求和、平均值、税额计算
3. 输出时保留两位小数，消除 `0.1 + 0.2` 类浮点误差

**影响评估**：需安装新依赖（`pnpm add decimal.js`），修改计算函数，需对比修改前后结果一致性

---

#### P2-C：工资条 — 智能字段匹配扩展

**改哪里**：`server/hr.ts`，`detectPayslipFields` 的 fallback 关键词列表

**具体做法**：
- 扩展 `baseSalaryCol` 匹配词：加入"固定工资"、"月薪"、"标准工资"
- 扩展 `bonusCol` 匹配词：加入"绩效工资"、"KPI奖金"、"季度奖"
- 扩展 `deductionCol` 匹配词：加入"缺勤扣款"、"违规扣款"

**影响评估**：仅扩展关键词列表，零风险

---

#### P2-D：考勤 — 加班计算

**改哪里**：`server/hr.ts`，`analyzeAttendance` 函数

**具体做法**：
1. 新增参数：`overtimeRateWeekday`（平日加班倍率，默认 1.5）、`standardHours`（标准工时，默认 8h）
2. 计算每条记录的实际工时 = 下班时间 - 上班时间
3. 超出 `standardHours` 的部分计为加班，按倍率计算加班费
4. 汇总报表新增"加班时长"和"加班费"两列

**影响评估**：新增计算逻辑，不修改现有考勤汇总结构，向后兼容

---

### Phase 3（后续迭代）

- 行政场景：员工信息智能匹配（姓名 → 部门 ID 关联）
- 运营场景：商品名称 NLP 标准化（SKU 归一化）
- HR 中心：对话框上方常驻快捷按钮区（工资条 / 考勤汇总）
- 全局异常拦截框架：负数收入、超大金额（>10万）自动标红

## 三、回归测试范围

每个 Phase 上线前，必须验证以下功能**未受影响**：

| 功能 | 测试方法 |
|------|---------|
| 普通文件上传 + AI 分析（sales/general 场景） | 上传一份销售表，确认 suggestedActions 正常展示，AI 分析结果正确 |
| 多文件上传 + 跨表合并 | 上传两份店铺数据，确认合并提示和汇总结果正确 |
| HR 工资条完整流程 | 上传工资表 → 字段识别 → 生成 Excel → 下载，全流程验证 |
| HR 考勤完整流程 | 上传考勤表 → 字段识别 → 生成汇总 → 下载，全流程验证 |
| AI 对话（无文件） | 直接发消息，确认 AI 正常回复 |
| 现有 69 个 vitest 测试 | `pnpm test` 全部通过 |

## 四、开发顺序建议

```
Phase 1（本周）
  ├─ P1-A：atlas.ts 字段标准化 + 缺失值容错（约 2h）
  └─ P1-B：工资条/考勤一键跳转（约 3h）

Phase 2（下周）
  ├─ P2-A：借贷平衡检查（约 2h）
  ├─ P2-B：decimal.js 高精度计算（约 3h）
  ├─ P2-C：工资条字段匹配扩展（约 0.5h）
  └─ P2-D：考勤加班计算（约 2h）

Phase 3（后续）
  └─ 行政/运营深度优化、HR 快捷按钮区
```
