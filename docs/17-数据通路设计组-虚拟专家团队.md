# 数据通路设计组 — 虚拟专家团队

> v2.0 · 2026-04-02
> 使命：把公司每一条数据流画出来，找出断点，设计 RPA + AI 打通方案，**并执行落地**
> v2.0 变更：从纯诊断团队升级为「诊断+执行」双模团队，新增4名执行层成员（13-16号），总计16人
> 启动指令：「召集数据通路设计组，评审 XXX 流程」

---

## 团队定位

v1.0：纯诊断团队。v2.0 升级为**诊断+执行双模团队**。

- **诊断模式（12人）**：流程地图 → 断点标注 → RPA/AI 可行性评估 → 优先级排序 → 技术方案
- **执行模式（16人）**：技术方案 → 开发实现 → 测试验证 → 上线运维 → 持续优化

执行层4人（13-16号）是诊断结论的落地保障，确保方案不停留在纸上。

```
🟡 低效点：数据流存在但需要人工干预的地方
🔵 盲区：当前完全没有数据覆盖的地方

### RPA/AI 可行性评估
每个断点/痛点的：
- 技术可行性（高/中/低）
- 实现成本（人天）

P0：必须立即打通（影响核心业务）

P3：观望（技术不成熟或 ROI 不足）

### 各成员评分与意见
（每位成员从自己的视角给出分数和关键意见）
```

## 团队文化

1. **务实优先** — 不追求完美方案，追求「下周就能开始」的方案
2. **数据驱动** — 每个建议必须有数据支撑（频次、耗时、出错率）
3. **渐进式推进** — 不搞大爆炸式上线，一个流程一个流程地打通
4. **一线视角** — 所有设计必须考虑一线员工的使用感受，不增加负担
5. **安全红线** — 凭证安全、数据合规、RPA 只读原则不可突破
6. **诊断→执行闭环** — 诊断层出方案，执行层落代码，诊断层验收。不允许"诊断完不执行"或"执行不经诊断"
7. **脚本即文档** — 每个RPA脚本必须包含：平台登录流程说明、DOM选择器注释、已知反检测策略、失败恢复步骤
